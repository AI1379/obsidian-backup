## 一、操作系统到底在做什么

### 1. OS 的本质

操作系统不是“一个大号工具库”，而更像：

> 硬件资源的管理者，以及给应用程序提供抽象接口的中间层。

它主要管理五类东西：

1. **CPU**
   - 谁先跑
   - 谁后跑
   - 谁阻塞
   - 谁被唤醒

2. **内存**
   - 哪块内存分给谁
   - 哪些页在主存
   - 哪些页换出去
   - 谁能访问哪些地址

3. **文件和外设**
   - 文件怎么组织
   - 网卡、磁盘、键盘怎么访问
   - IO 完成后怎么通知 CPU

4. **进程和线程**
   - 怎么创建
   - 怎么退出
   - 怎么通信
   - 怎么同步

5. **权限与隔离**
   - 用户程序能做什么
   - 哪些事必须进内核
   - 进程之间如何隔离

### 2. 为什么需要操作系统

没有 OS，应用程序就得自己：

- 管 CPU
- 管内存
- 管磁盘
- 管网卡
- 管中断

这几乎不可维护。
OS 的意义就是把复杂硬件能力抽象成统一接口，例如：

- 文件
- 进程
- 线程
- socket
- 虚拟地址空间

### 3. 用户态和内核态为什么存在

如果任何程序都能直接：

- 改页表
- 关中断
- 写磁盘控制器
- 读别人内存

系统就会非常不安全。

因此 OS 通常引入：

#### 用户态

- 权限低
- 运行普通应用逻辑
- 不能做特权操作

#### 内核态

- 权限高
- 能访问硬件、调度、内存管理、处理中断

### 4. 系统调用是什么

系统调用可以理解成：

> 用户程序向操作系统申请帮忙完成一件需要特权的事情。

例如：

- `open`
- `read`
- `write`
- `fork`
- `mmap`
- `epoll_wait`

一个直觉流程是：

1. 用户态代码发起系统调用
2. CPU 切到内核态
3. 内核检查参数和权限
4. 内核执行服务
5. 返回用户态

> **"CPU 切到内核态"到底是什么意思？**
>
> 用户态和内核态不只是 OS 层面的概念，它首先是 **CPU 硬件提供的特权级机制**。
> 比如 x86 有 Ring 0 ~ Ring 3，ARM 有 EL0 ~ EL3，CPU 内部有一个"当前特权级"标志。
> Ring 0（内核态）能执行所有指令，Ring 3（用户态）不能执行特权指令（关中断、写页表、直接访问硬件等）。
>
> OS 内核运行在 Ring 0，用户程序运行在 Ring 3。
> 系统调用时，用户程序执行 `syscall` 等指令，触发一个 **trap/中断**，
> **CPU 硬件自动**把特权级从 Ring 3 切到 Ring 0 并跳转到内核入口。
> 这个切换是硬件完成的，不是 OS 自己用软件改个标志位——否则用户程序也能伪造切换。
>
> 所以"CPU 切到内核态"准确地说：**通过硬件 trap 机制，CPU 自动提升特权级，进入内核代码执行。**

### 5. 为什么面试里 OS 和并发经常混着问

因为并发问题很多时候不是纯语言层面问题，而会牵扯：

- 线程调度
- 阻塞与唤醒
- 用户态 / 内核态切换
- 文件描述符
- epoll
- 虚拟内存

所以如果你只会背“锁 API”，但不知道它和 OS 怎么配合，面试里会显得浅。

## 二、进程、线程、协程

### 1. 进程是什么

进程是：

> 操作系统进行资源分配的基本单位。

它通常拥有：

- 独立地址空间
- 打开的文件
- 代码段、数据段、堆、栈
- 各类资源句柄

### 2. 线程是什么

线程是：

> 进程里的执行流，是 CPU 调度的基本单位。

同一进程内多个线程通常共享：

- 地址空间
- 堆
- 全局变量
- 文件描述符

但各自有：

- 栈
- 寄存器上下文
- 程序计数器

### 2.1 硬件线程 vs 软件线程

> 买 CPU 时常说的"8 核 16 线程"里的"线程"，和 OS 里的"线程"不是同一个东西。

#### 硬件线程（Hardware Thread / Logical Core）

- 对应 CPU 的**物理执行能力**
- 一个物理核通过 **SMT（超线程 / Hyper-Threading）** 可以假装成两个逻辑核
- 比如一块 8 核 16 线程的 CPU，有 8 个物理核，但 OS 能看到 16 个逻辑核（硬件线程）
- 硬件线程的个数是**固定的、有限的**，由 CPU 硬件决定

#### 软件线程（Software Thread / OS Thread）

- OS 管理的**执行流**，就是上面说的"进程里的线程"
- 数量可以有很多（Linux 默认上限可以上万）
- OS 的调度器负责把软件线程**分配到硬件线程上执行**
- 软件线程数 >> 硬件线程数，所以需要调度和时间片轮转

#### 一句话关系

> 硬件线程是"能同时跑几条执行流的物理资源"，软件线程是"OS 管理的逻辑执行流"。
> OS 调度器把成百上千的软件线程，映射到有限的硬件线程上轮流执行。

#### 面试怎么答

"CPU 说的线程数是硬件线程（逻辑核数），代表物理上能并行执行的流数；
OS 说的线程是软件线程，是调度管理的逻辑执行流。
两者是多对一的映射关系——很多软件线程被 OS 调度到少量硬件线程上轮流执行。"

### 3. 为什么线程比进程轻

因为线程不需要一整套独立地址空间。
所以：

- 创建更轻
- 切换通常更轻
- 通信更方便

但共享内存也会带来同步问题。

### 4. 协程是什么

协程可以理解为：

> 用户态调度的轻量执行单元。

特点：

- 切换成本通常比线程低
- 很适合高并发 IO
- 需要运行时或框架支持

### 5. 进程和线程的状态

最常见的直觉状态：

- 就绪
- 运行
- 阻塞 / 等待
- 终止

可以这样记：

- **就绪**：我能跑，但还没轮到我
- **运行**：我正在占用 CPU
- **阻塞**：我现在跑也没用，我在等 IO / 锁 / 条件

### 6. fork / exec / wait

这是 OS 八股高频三件套。

#### fork

创建子进程。

#### exec

用新程序替换当前进程的地址空间。

#### wait

父进程等待子进程结束并回收其退出状态。

#### 一句话理解

> fork 造新进程，exec 换程序，wait 负责等和收尸。

### 7. 僵尸进程和孤儿进程

#### 僵尸进程

子进程已经退出，但父进程还没 `wait`，导致退出信息还残留在系统里。

特点：

- 不再运行
- 但仍占少量内核表项

#### 孤儿进程

父进程先退出了，子进程还活着。
通常会被新的父进程接管并最终回收。

#### 高频误区

- 僵尸进程不是“卡死运行中的进程”
- 它已经死了，只是还没被回收

### 8. IPC：进程间通信

因为进程地址空间隔离，不能像线程那样直接共享普通变量。
所以需要 IPC。

常见方式：

- pipe / 管道
- message queue
- shared memory
- socket
- signal

#### 粗略怎么选

- 同机高性能通信：共享内存常见
- 跨机通信：socket 更自然
- 父子进程简单流式数据：管道很常见

### 9. 游戏后端里怎么选

#### 进程

适合：

- 服务隔离
- 模块隔离
- 崩溃隔离

#### 线程

适合：

- 利用多核
- 并行任务处理

#### 协程

适合：

- 大量网络 IO
- 数据库 IO
- 异步逻辑编排

#### 面试版表达

> 线程更适合 CPU 并行和共享内存场景，协程更适合高并发 IO 场景；实际服务端工程里常见的是多线程 + 协程结合。

## 三、调度、上下文切换、用户态和内核态

### 1. 为什么需要调度

CPU 核心数有限，但线程数往往很多。
OS 必须决定：

- 现在谁跑
- 谁等待
- 谁被唤醒

### 2. 上下文切换切的是什么

线程切换时，系统需要保存和恢复：

- 寄存器
- 程序计数器
- 栈指针
- 调度状态
- 部分内核态信息

### 3. 为什么频繁切换会慢

因为切换本身是纯额外开销：

- 不产生业务价值
- 会破坏 Cache 局部性
- 会增加调度成本

### 4. 常见调度算法

#### FCFS

先来先服务。
简单，但短任务可能被长任务拖住。

#### RR

时间片轮转。
每个任务轮流拿时间片，适合交互系统的公平推进。

#### 优先级调度

优先执行高优先级任务。
但低优先级任务可能饥饿。

#### 多级反馈队列

很经典的工程思路：

- 新任务先高优先级
- CPU 密集型任务逐步下沉
- 交互型任务更容易快速响应

#### 面试要点

不要只背名字，要知道它们在权衡：

- 公平性
- 响应时间
- 吞吐

### 5. 为什么锁会和内核态扯上关系

像 mutex 在竞争激烈时，线程可能需要：

- 阻塞
- 被内核挂起
- 再被内核唤醒

所以锁并不只是一个“语言级关键字”。

## 四、内存管理与虚拟内存

### 1. 内存管理到底在管什么

OS 要解决的是：

- 哪块物理内存分给谁
- 谁能访问哪里
- 不够用时怎么办
- 程序如何看到“连续、独立”的地址空间

### 2. 分页和分段

#### 分页

把地址空间切成固定大小的页。

优点：

- 管理方便
- 外部碎片少

#### 分段

按逻辑意义划分，例如：

- 代码段
- 数据段
- 栈段

优点是更符合程序逻辑，但管理更复杂。

#### 面试表达

现代系统通常更偏分页，分段更多作为逻辑概念或与分页结合存在。

### 3. 虚拟内存

虚拟内存的核心价值：

1. 给每个进程独立地址空间
2. 提高内存利用率
3. 支持按需调页
4. 实现隔离与保护

### 4. 页表、TLB、缺页异常

#### 页表

记录虚拟页到物理页框的映射。

#### TLB

缓存最近常用页表项，加速地址翻译。

#### 缺页异常

访问的页不在物理内存里时，进入内核，由 OS 从外存调页。

### 5. 页面置换算法

#### FIFO

谁先进来谁先出去。
实现简单，但效果可能一般。

#### LRU

最近最少使用。
符合局部性直觉，但精确实现成本高。

#### Clock

常作为 LRU 的工程近似。

#### 高频点

面试里常考的是：

- 这些算法的直觉
- 为什么 LRU 思想合理
- 为什么真实系统常用近似算法

### 6. 抖动和工作集

#### 工作集

某段时间内程序频繁访问的页集合。

#### 抖动

如果物理内存太小，工作集装不下，系统就会频繁换页。
CPU 大量时间花在缺页和换页上，而不是干活。

### 7. mmap 和共享内存直觉

`mmap` 可以把文件或内存对象映射到进程地址空间。
它常用于：

- 高效文件 IO
- 共享内存
- 零拷贝相关优化

## 五、文件系统、系统调用与 IO 基础

### 1. 文件系统到底在解决什么

文件系统解决的是：

- 数据如何持久化
- 路径如何组织
- 文件如何定位
- 崩溃后怎样尽量保证一致性

### 2. inode、目录、数据块

#### inode

记录文件元数据，例如：

- 权限
- 大小
- 时间戳
- 数据块位置

#### 目录

目录更像“文件名到 inode 的映射表”。

#### 数据块

真正存文件内容的地方。

#### 高频点

文件名通常不直接放在 inode 里，而是目录项负责名字映射。

### 3. 硬链接和软链接

#### 硬链接

多个名字指向同一个 inode。

#### 软链接

一个特殊文件，里面保存目标路径。

#### 高频区别

- 硬链接更像同一个文件的另一个名字
- 软链接更像快捷方式

### 4. 文件读写的大致路径

一个简化流程：

1. 应用调用 `read` / `write`
2. 进入内核
3. 查文件描述符
4. 查页缓存 / 文件系统元数据
5. 必要时访问磁盘
6. 把数据返回用户态或写入缓存

### 5. 阻塞 / 非阻塞、同步 / 异步

这组特别容易混。

#### 阻塞 / 非阻塞

强调的是：

> 当前调用会不会把调用方卡住。

#### 同步 / 异步

强调的是：

> 完成这件事时，调用方要不要自己一直盯着。

#### 面试直觉

- 阻塞/非阻塞：看“你卡不卡”
- 同步/异步：看“你要不要亲自盯到完成”

### 6. 零拷贝、sendfile、mmap

零拷贝不是完全没有任何复制，更准确是：

> 尽量减少用户态和内核态之间不必要的数据来回搬运。

#### sendfile

常用于高效发送文件，减少用户态中转。

#### mmap

通过映射减少部分显式拷贝和系统调用开销。

### 7. fsync 为什么会慢

因为 `fsync` 常要求：

- 把缓存中的脏数据真正刷到稳定存储

这会逼迫系统和磁盘做更重的持久化动作，所以延迟会更高。

## 六、并发到底在解决什么问题

### 1. 单线程为什么不够

单线程简单，但它有明显局限：

- 一个慢 IO 可能拖住整个流程
- 无法充分利用多核 CPU
- 高并发连接场景下吞吐可能上不去

### 2. 并发和并行的区别

#### 并发

一段时间内，多个任务都在推进。
不一定同一时刻真正在多个核上跑。

#### 并行

同一时刻，多个任务真的同时执行。

#### 面试一句话

- 并发更强调结构上同时处理多个任务
- 并行更强调物理上同时执行

### 3. 游戏后端为什么天然会遇到并发

因为它通常同时处理：

- 大量玩家连接
- 登录、匹配、房间、战斗、聊天、结算
- 定时器、网络 IO、数据库 IO、RPC 调用

你很难指望一条完全串行的执行流就把所有这些优雅处理完。

## 七、锁与同步原语

### 1. 竞态条件、临界区、互斥与同步

这四个是**并发编程的基本概念**，不是死锁的四个必要条件（后面会专门讲死锁）。

#### 竞态条件（Race Condition）

多个线程访问同一份共享数据，**结果的正确性依赖线程的执行时序**，这就是竞态条件。

例如：

```text
x = x + 1
```

看起来是一行，但底层常是：

1. 读 x
2. 加 1
3. 写回 x

如果两个线程同时执行，可能出现：
- 线程 A 读到 x = 5
- 线程 B 也读到 x = 5
- 线程 A 写回 6
- 线程 B 也写回 6

本应该是 7，结果变成了 6。这就是竞态条件导致的 bug。

> 本质：**"谁先谁后"会影响结果的正确性，但你控制不了谁先谁后。**

#### 临界区（Critical Section）

访问共享资源的那段**必须谨慎控制**的代码。

- 它不是一个锁，而是一个**代码区域**
- 临界区里做的事情，不能让多个线程同时执行
- 比如上面 `x = x + 1` 所在的代码段就是临界区

> 本质：**"这段代码出问题的区域"——竞态条件发生的场所。**

> 参考 jyy 的网课：这里就是一段需要用 `__sync_synchronize()` 或者至少是 `BARRIER` 包裹起来的代码块。
#### 互斥（Mutual Exclusion）

**同一时刻只允许一个线程进入临界区。**

- 互斥是一种**保护策略**
- 实现方式：mutex、spinlock 等锁
- 效果：保证不会有多个线程同时在临界区里操作共享数据

> 本质：**"我进去了你就不能进"——解决竞态条件的基本手段。**

#### 同步（Synchronization）

保证线程之间**按正确的顺序协作**。

- 互斥解决的是"能不能同时进"的问题
- 同步解决的是"谁先谁后"的问题
- 典型场景：线程 A 必须等线程 B 做完某件事才能继续
- 实现方式：condition variable、barrier、semaphore 等

> 本质：**"你先做完我再做"——协调线程之间的执行顺序。**

#### 四者关系一句话

> 竞态条件是**问题**（结果依赖时序），临界区是**出问题的代码区域**，
> 互斥是解决"不能同时进"的**手段**，同步是解决"谁先谁后"的**手段**。

#### ⚠️ 注意：这不是死锁的四个必要条件

死锁的四个必要条件是**另一组完全不同的东西**：

| 死锁四个必要条件    | 本节四个并发概念 |
| ----------- | -------- |
| 1. 互斥（资源独占） | 竞态条件     |
| 2. 持有并等待    | 临界区      |
| 3. 不可剥夺     | 互斥       |
| 4. 循环等待     | 同步       |

虽然两边都有"互斥"这个词，但含义侧重不同：
- 死锁里的"互斥"强调的是**资源本身具有排他性**（一把锁同一时刻只能被一个线程持有）
- 本节的"互斥"强调的是**一种保护策略**（保证临界区同时只有一个线程进入）

两组是完全不同的分类，不要搞混。

### 1.5 CAS 前置知识

> 下面讲各种锁的算法时都会用到 CAS，所以先在这里解释清楚。

#### CAS 是什么

CAS = **Compare-And-Swap**（比较并交换）。它是 CPU 提供的一条**原子指令**。

它做的事情用伪代码表示：

```text
CAS(addr, expected, new_value):
    原子地执行：
        if *addr == expected:
            *addr = new_value
            return 成功
        else:
            return 失败
```

三个参数：
- `addr`：要改的内存地址
- `expected`：你预期这个地址现在应该存的值
- `new_value`：你想写进去的新值

#### 为什么 CAS 是原子的？

"比较 + 赋值"看起来是两步，但 CPU 的 CAS 指令在**硬件层面**保证这两步不可分割——没有其他指令能插在中间。这就是它能用来实现锁的基础。

x86 上对应 `cmpxchg` 指令（配合 `lock` 前缀），ARM 上对应 `LDXR/STXR` 或 `CAS` 指令。

#### CAS 怎么用来做锁？

思路很简单：

```text
// 加锁
lock:
    while CAS(&flag, 0, 1) 失败:
        // 没拿到，重试
    // 拿到了

// 解锁
unlock:
    flag = 0
```

- flag 初始为 0（未锁定）
- 加锁时 CAS 尝试把 0 改成 1
- 只有一个人能成功（因为 CAS 是原子的）
- 解锁时把 flag 改回 0

#### CAS vs 普通的 if + 赋值

为什么不能这样？

```text
if (flag == 0) {    // 第一步：读
    flag = 1;       // 第二步：写
}
```

因为第一步和第二步之间可能被打断：
- 线程 A 读到 flag == 0
- 线程 B 也读到 flag == 0
- 两人都进入 if，都写了 flag = 1
- 结果两个线程都以为自己拿到了锁 → 竞态条件

CAS 把"比较和写入"合成一条不可分割的原子操作，所以不会出现这个问题。

> 后面的 spinlock、mutex 底层都依赖 CAS。（CAS 更深入的内容在 [第八章](#八cas内存可见性happens-before死锁)）

### 2. spinlock（自旋锁）

**最底层的锁，算法也最简单。**

#### 算法

```
lock:
    while CAS(&flag, 0, 1) 失败:
        // 什么都不做，原地转圈（自旋）
    // CAS 成功，拿到锁

unlock:
    flag = 0   // 释放
```

核心就是**一个原子变量 + 一个死循环 CAS**。没拿到就一直在那转，不停重试。

#### 特点

- **纯用户态**，不涉及系统调用，不涉及内核
- 拿锁和释放锁的成本极低（就是一条原子指令）
- 等待期间**不释放 CPU**，一直占着核心空转
- 适合：临界区极短（几条指令）、持锁时间远小于一次上下文切换的开销
- 不适合：临界区长、可能在锁里做 IO、用户态长时间自旋

#### 为什么不能在用户态长时间自旋？

自旋期间线程是**运行状态**，一直占着 CPU 不干活。如果持锁时间长，其他线程自旋等待就是纯浪费 CPU。还不如让出 CPU 给有用的线程。

> 一句话：**spinlock = 原子 CAS 死循环。最轻，但不让出 CPU。**

---

### 3. mutex（互斥锁）

**最常见的锁，面试最高频。**

#### 算法（现代 Linux 的 pthread_mutex 实现）

现代 mutex（如 Linux glibc 的 `pthread_mutex`）不是简单的一把锁，而是一个**分层设计**：

```
lock:
    第一步：先尝试 CAS(&state, 0, 1)
        如果成功 → 直接拿到锁，返回（用户态完成，极快）

    第二步：CAS 失败，说明有竞争
        把 state 改成"可能有等待者"（值为 2）
        调用 futex_wait() 进入内核等待队列
        被唤醒后再尝试 CAS

unlock:
    把 state 减 1（原子操作）
    如果 state != 0，说明还有等待者
        调用 futex_wake() 唤醒一个等待线程
```

#### 关键：futex 机制

futex = **Fast Userspace muTEX**，是 Linux 提供的一个系统调用。

核心思想：

- **无竞争时**：完全在用户态用 CAS 完成，**零系统调用**
- **有竞争时**：通过 `futex_wait` 让线程在内核中**真正睡眠**（不占 CPU），通过 `futex_wake` 唤醒

这就是为什么 mutex 比 spinlock "重"——竞争时需要进内核、做上下文切换，但好处是**不浪费 CPU**。

#### 特点

- 抢不到时线程**阻塞（睡眠）**，让出 CPU 给其他线程
- 适合：临界区较长、可能等待的场景
- 代价：无竞争时很轻（一次 CAS），有竞争时需要系统调用 + 上下文切换

> 一句话：**mutex = CAS + futex。无竞争快如自旋，有竞争睡眠让 CPU。**

---

### 4. rwlock（读写锁）

**在 mutex 基础上区分读写，允许并发读。**

#### 算法思路

内部维护：

- 一个**读者计数器**（当前有多少读者在临界区里）
- 一个**写者标志**（当前是否有写者在临界区里）

```
read_lock:
    while 写者标志 == 1:
        等待（自旋或睡眠，取决于实现）
    读者计数器 += 1（原子操作）

read_unlock:
    读者计数器 -= 1
    如果读者计数器 == 0 且有写者在等:
        唤醒一个写者

write_lock:
    while 写者标志 == 1 或 读者计数器 > 0:
        等待
    写者标志 = 1

write_unlock:
    写者标志 = 0
    唤醒等待的读者或写者
```

#### 关键问题：写者饥饿

如果读者源源不断，读者计数器一直 > 0，写者就可能**永远等下去**。

所以实际实现会有一些策略来防止写者饥饿，比如：
- 有写者在等待时，后续读者不能再进入（写者优先策略）
- 或者用公平队列，按到达顺序排

#### 特点

- 适合：读多写少的场景
- 不一定总比 mutex 快——如果读写都频繁，rwlock 内部的计数和判断开销可能超过直接用 mutex
- 注意写者饥饿问题

> 一句话：**rwlock = mutex + 读者计数器。读可以并行，写必须独占。**

---

### 5. semaphore（信号量）

**带计数器的同步原语，比 mutex 更通用。**

#### 算法

信号量内部就是一个**整数计数器** + **等待队列**：

```
wait(P 操作):
    计数器 -= 1（原子操作）
    如果计数器 < 0:
        把当前线程加入等待队列，阻塞

signal(V 操作):
    计数器 += 1（原子操作）
    如果等待队列非空:
        唤醒一个等待线程
```

#### 和 mutex 的关系

- **mutex 本质上就是计数器初始值为 1 的信号量**（二元信号量 / binary semaphore）
- 但 mutex 有一个额外语义：**谁加锁谁解锁**（所有权）
- 信号量没有所有权限制——任何线程都可以 signal

#### 常见用途

- **限制并发数**：比如初始化为 100，最多允许 100 个线程同时进入
- **生产者-消费者模型**：一个信号量跟踪"空闲槽位"，一个信号量跟踪"已填充槽位"

> 一句话：**semaphore = 原子计数器 + 等待队列。mutex 是它的特例（计数 = 1）。**

---

### 6. condition variable（条件变量）

**不是锁，是用来等"条件成立"的同步机制。**

#### 为什么需要它？

想一下这个场景：

```text
while (队列为空) {
    // 怎么等？sleep？自旋？
}
从队列取元素
```

- 自旋：浪费 CPU
- sleep：等多久？不确定

条件变量就是解决这个问题的。

#### 算法（必须和 mutex 配合使用）

```text
消费者：
    mutex.lock()
    while (条件不满足) {
        cv.wait(mutex)    // 原子地：释放 mutex + 进入等待
                          // 被唤醒时：重新获取 mutex + 返回
    }
    // 条件满足了，干活
    mutex.unlock()

生产者：
    mutex.lock()
    // 改变条件（比如往队列加元素）
    cv.signal() 或 cv.broadcast()   // 唤醒一个或所有等待者
    mutex.unlock()
```

#### 为什么 wait 里要用 while 不用 if？

因为存在**虚假唤醒（spurious wakeup）**：线程可能在没有被 signal 的情况下被唤醒。
所以必须在唤醒后再检查一次条件。用 while 就能自然处理这种情况。

#### 特点

- 不是锁，不保护临界区，而是**等条件**
- 必须和 mutex 配合（否则条件本身的读写也是竞态）
- signal 唤醒一个，broadcast 唤醒所有

> 一句话：**cv = "条件不满足就睡觉，满足了我来叫你"。必须配 mutex 使用。**

---

### 7. 各种锁的对比总览

| 锁 | 底层算法 | 等待方式 | 适用场景 |
|---|---|---|---|
| spinlock | CAS 死循环 | 自旋（不释放 CPU） | 极短临界区、内核态常见 |
| mutex | CAS + futex | 睡眠（让出 CPU） | 通用场景、临界区较长 |
| rwlock | mutex + 读者计数器 | 睡眠 | 读多写少 |
| semaphore | 原子计数器 + 等待队列 | 睡眠 | 限并发、生产者消费者 |
| condition variable | 等待队列（配 mutex） | 睡眠 | 等条件成立 |
| futex | 不是锁，是 Linux 系统调用 | — | mutex 的底层基础设施 |

### 8. 怎么选锁

- 临界区短、竞争轻：可考虑自旋
- 临界区长、可能阻塞：更偏 mutex
- 读多写少：可考虑 rwlock
- 需要等条件：condition variable
- 需要限并发：semaphore

## 八、CAS、内存可见性、happens-before、死锁

### 1. CAS 是什么

CAS = Compare-And-Swap。

思路：

1. 比较当前值是不是预期值
2. 如果是，就改成新值
3. 如果不是，说明有人改过，失败重试

### 2. CAS 和 mutex 的区别

#### mutex

- 悲观
- 先排他，再改

#### CAS

- 乐观
- 先尝试，失败再重试

### 3. ABA 问题

某个值看起来还是 `A`，不代表它没被改过。
它可能经历过：

- `A -> B -> A`

这就是 ABA。

### 4. CAS 的代价

竞争激烈时：

- 会大量失败重试
- 会消耗 CPU
- 还会加剧缓存一致性流量

### 5. 为什么一个线程写了，另一个线程未必马上看到

原因可能包括：

- 值还在寄存器
- 值还在本核 Cache
- 编译器重排
- CPU 重排

### 6. 内存屏障的直觉

内存屏障可以理解成：

> 在某些关键点，不允许某些读写越过去。

它在帮助保证：

- 顺序
- 可见性

### 7. happens-before 怎么理解

它不是简单的“时间先后”，而更像：

> 一套保证：如果 A happens-before B，那么 A 的结果对 B 可见，且顺序被认可。

很多同步原语都在帮你建立这种关系。

### 8. 死锁、活锁、饥饿

#### 死锁（Deadlock）

**多个线程互相持有对方需要的资源，又互相等待对方释放，形成循环依赖，谁也走不下去。**

##### 用图论来理解

死锁本质上就是一个**资源分配图（Resource Allocation Graph）** 里的环。

把系统状态画成一张有向图：
- **节点**有两类：进程（线程）节点 和 资源节点
- **边**有两类：
  - 进程 → 资源：表示”进程正在**请求**这个资源”（请求边）
  - 资源 → 进程：表示”资源被这个进程**持有**”（分配边）

```
线程 A                      线程 B
  │                          │
  │ 持有锁 L1                 │ 持有锁 L2
  ▼                          ▼
 锁 L1 ───请求───→ 锁 L2 ───请求───→ 锁 L1
         (B 等待 L1)         (A 等待 L2)
```

当这张图里出现**环（循环）**时，就是死锁：
- 线程 A 持有 L1，等待 L2
- 线程 B 持有 L2，等待 L1
- A 等 B，B 等 A → 环 → 死锁

> **判断是否死锁 = 判断资源分配图里有没有环。**
> 如果每种资源只有一个实例，图中有环 **等价于** 死锁。
> 如果某种资源有多个实例，有环是死锁的**必要条件**但不一定充分（可能环上的某个请求刚好能被满足）。

##### 四个必要条件（缺一不可）

死锁发生**必须同时满足**这四个条件。反过来，**打破其中任何一个就能预防死锁**：

**1. 互斥（Mutual Exclusion）**

> 资源本身具有排他性，同一时刻只能被一个线程使用。

比如一把锁，同一时刻只能被一个线程持有。如果一个资源大家都能同时用，就不存在等待，就不会死锁。

**2. 持有并等待（Hold and Wait）**

> 线程已经持有了至少一个资源，同时又在等待其他资源，且**不释放**已持有的资源。

比如线程 A 拿了锁 L1，再去等锁 L2，但手里一直攥着 L1 不放。如果线程在请求新资源前先把已有的都释放了，就不会死锁。

**3. 不可剥夺（No Preemption）**

> 线程持有的资源不能被强制抢走，只能由线程自己释放。

操作系统不能说”你占太久了，我强制把锁收回”。如果可以强制剥夺，被等的人就会被赶走，环就断了。

**4. 循环等待（Circular Wait）**

> 存在一个线程的循环链：A 等 B，B 等 C，……，最后一个等 A。

这就是图论里的环。如果等待关系不成环，就不会死锁。

##### 四个条件的总结

| 条件 | 含义 | 打破它的思路 |
|---|---|---|
| 互斥 | 资源排他，不能共享 | 让资源可共享（不总是可行） |
| 持有并等待 | 拿着一个，再去等另一个 | 一次拿完所有锁再开始，或者拿不到就全释放 |
| 不可剥夺 | 资源不能被强制回收 | 超时机制，拿不到就放弃已有的 |
| 循环等待 | 等待关系形成环 | **固定加锁顺序**，破坏成环的可能 |

---

#### 活锁（Livelock）

**线程们没有卡死，都在”积极响应”，但因为互相避让或者不断重试，谁也完不成实际工作。**

##### 经典比喻

走廊里两个人迎面走来：
- A 往左让，B 也往左让 → 还是挡着
- A 往右让，B 也往右让 → 还是挡着
- 如此反复……

两人都在动，都没有停下来（不是死锁），但永远过不去。

##### 代码里的活锁

```text
线程 A:
    while true:
        if trylock(L1) 成功:
            if trylock(L2) 成功:
                做事
                break
            else:
                unlock(L1)     // 拿不到 L2，放弃 L1
                sleep(随机时间) // 避让一下
                重试

线程 B:（对称）
    while true:
        if trylock(L2) 成功:
            if trylock(L1) 成功:
                做事
                break
            else:
                unlock(L2)
                sleep(随机时间)
                重试
```

如果 A 和 B 的随机时间总是接近（或者同步递增），它们可能反复在同一时刻重试，互相撞车，永远拿不到两把锁。

##### 活锁 vs 死锁

| | 死锁 | 活锁 |
|---|---|---|
| 线程状态 | **阻塞/睡眠**，什么都不做 | **运行中**，不断重试 |
| CPU 占用 | 不占 CPU | 浪费 CPU |
| 能否自愈 | 不能，必须外部干预 | 理论上有可能自愈，但实际很难 |
| 本质 | 等待环，彻底卡住 | 互相避让，反复冲突 |

---

#### 饥饿（Starvation）

**某个线程长期拿不到资源，一直被”插队”，始终无法推进。**

##### 常见场景

- rwlock 中，读者源源不断，写者永远等不到（**写者饥饿**）
- 优先级调度中，低优先级线程永远被高优先级线程抢占
- 锁的实现中，新来的线程总是比等待很久的线程先拿到锁（**非公平锁**）

##### 饥饿 vs 死锁

- 死锁：所有涉及的人都卡住了
- 饥饿：只有部分人被”饿着”，其他人可能正常推进

---

### 9. 怎么避免死锁、活锁、饥饿

#### 避免死锁

**方法 1：固定加锁顺序（最常用、最实用）**

规定所有线程必须按**固定顺序**获取锁。比如总是先锁 L1 再锁 L2，永远不会出现”A 拿 L1 等 L2，B 拿 L2 等 L1”的环。

```text
// 正确：所有线程都按 L1 → L2 的顺序加锁
lock(L1)
lock(L2)
// 干活
unlock(L2)
unlock(L1)
```

这本质上就是**打破”循环等待”条件**——如果所有线程都按同一个顺序拿锁，等待关系就是一条链，不可能成环。

**方法 2：trylock + 回退**

用 `trylock`（非阻塞尝试），拿不到就放弃已有的锁，等一会再重来。

```text
lock(L1)
if trylock(L2) 成功:
    // 干活
    unlock(L2)
    unlock(L1)
else:
    unlock(L1)        // 放弃 L1
    sleep(随机时间)    // 等一会再重试
```

这打破了 **”持有并等待”** 条件——拿不到全部就全部放弃。

**方法 3：锁层级设计**

给锁分层，规定只能从高层向低层加锁。比如：
- 层级 0：全局配置锁
- 层级 1：模块锁
- 层级 2：对象锁

只能从 0 → 1 → 2 加锁，不能反向。等价于给锁定义了一个全序关系，不可能成环。

**方法 4：减少锁嵌套**

能只用一把锁解决的，不要用两把。锁越少，成环的可能性越低。

**方法 5：用消息队列代替共享状态锁**

如果线程之间通过消息传递而不是共享内存来通信，就不需要锁，自然不会有死锁。这就是 actor 模型的思路（后面第十章会讲）。

#### 避免活锁

**方法 1：随机退避**

重试前等待一个**随机时间**，而不是固定时间。这样两个线程不太可能反复同步冲突。

```text
sleep(random(1ms, 100ms))  // 随机退避
```

**方法 2：指数退避**

每次冲突后，等待时间翻倍：1ms → 2ms → 4ms → 8ms……逐渐拉开重试间隔。

**方法 3：引入优先级或协调机制**

让某个线程”先走”，而不是大家都谦让。比如让线程 ID 小的先尝试。

#### 避免饥饿

**方法 1：使用公平锁**

公平锁按**先来后到**的顺序分配，等待最久的线程优先拿到。比如 Linux 的 `pthread_mutex` 可以设置为 `PTHREAD_MUTEX_FAIR` 或使用 MCS 队列锁。

**方法 2：设置优先级或配额**

确保低优先级线程也能定期获得执行机会。比如：
- rwlock 中加入”写者优先”策略，防止写者饥饿
- 调度器中引入老化（aging）机制：等得越久优先级越高

**方法 3：避免无限竞争**

如果某个操作可能被无限次抢占，考虑加一个上限或保护机制。

#### 三者避免策略总结

| 问题 | 核心原因 | 最有效的避免手段 |
|---|---|---|
| 死锁 | 循环等待 | 固定加锁顺序 |
| 活锁 | 互相避让、同步冲突 | 随机/指数退避 |
| 饥饿 | 不公平调度 | 公平锁 / 老化机制 |

## 九、IO 多路复用与事件驱动

### 1. 为什么需要 IO 多路复用

高并发网络服务里，不能给每个连接都配一个线程去阻塞等待。
那样线程数会爆炸。

所以需要一种机制：

> 一个线程同时盯很多连接，谁有事件就处理谁。

**朴素做法的问题：**

```text
方案 1：每个连接一个线程，阻塞 read
  → 1 万个连接 = 1 万个线程 → 内存爆炸 + 调度爆炸

方案 2：非阻塞轮询，for 循环挨个检查每个 fd
  → 可以，但每次要遍历所有 fd，大部分没有数据，白忙

方案 3：IO 多路复用
  → 内核帮你盯，只告诉你有事件的 fd → 高效
```

### 2. select

**最早的 IO 多路复用机制， POSIX 标准，几乎所有系统都支持。**

#### 工作方式

```text
select(maxfd, readfds, writefds, exceptfds, timeout):
    1. 把用户传入的 fd 集合（bitmap）复制到内核
    2. 内核遍历所有 fd，检查哪些有事件
    3. 把结果写回 bitmap，返回就绪的 fd 数量
    4. 用户态再遍历 bitmap，找到哪些 fd 就绪了
```

#### 核心问题

**1. fd 数量有硬上限**

- `FD_SETSIZE` 通常是 1024（编译时常量）
- 要改的话要重新编译程序
- 对于高并发服务（几万连接）完全不够

**2. 每次调用都要全量传递 + 全量扫描**

- 每次调用 select，都要把**所有 fd 的集合**从用户态复制到内核态
- 内核要**线性遍历**所有 fd 检查状态
- fd 很多但只有少数活跃时，绝大多数工作是白做的

**3. 返回后用户还要线性遍历**

- select 只告诉你”有几个 fd 就绪了”，不告诉你**是哪几个**
- 你必须遍历整个 fd 集合逐个检查

#### 性能特征

| 连接数 | 就绪数 | select 的工作量 |
|---|---|---|
| 10 | 3 | 扫描 10，返回遍历 10 |
| 1000 | 3 | 扫描 1000，返回遍历 1000 |
| 1024（上限） | 3 | 扫描 1024，返回遍历 1024 |

> 连接越多、活跃率越低，select 越浪费。

---

### 3. poll

**对 select 的改进，去掉了 fd 数量的硬上限。**

#### 工作方式

```text
poll(fds[], nfds, timeout):
    1. 用户传入一个 struct pollfd 数组（每个元素含 fd、关注的事件、返回的事件）
    2. 内核遍历数组，检查每个 fd
    3. 有事件的标记返回，用户遍历数组检查
```

#### 相比 select 的改进

- **没有 1024 的硬上限**：用动态数组代替了固定 bitmap
- **接口更清晰**：用结构体数组代替 bitmap，不用位操作

#### 核心问题（没解决的）

- **仍然要线性遍历所有 fd**：内核侧和用户侧都是 O(n)
- **每次调用都要全量传递**：所有 fd 信息每次都要从用户态复制到内核态

#### select vs poll

| | select | poll |
|---|---|---|
| fd 上限 | 1024（`FD_SETSIZE`） | 无硬上限（受系统资源限制） |
| 数据结构 | bitmap | 结构体数组 |
| 内核扫描方式 | 线性遍历 | 线性遍历 |
| 用户检查方式 | 遍历 bitmap | 遍历数组 |
| 时间复杂度 | O(n) | O(n) |

> 本质上 poll 只是”更好用的 select”，核心性能瓶颈（O(n) 遍历）没有解决。

---

### 4. epoll

**Linux 专有的 IO 多路复用机制，解决 select/poll 的 O(n) 问题。**

#### 核心思路：把”每次遍历”变成”只返回就绪的”

select/poll 的问题是：每次调用都要把**所有 fd** 传进去，内核**遍历所有 fd**。

epoll 的做法完全不同：

```text
// 第一步：创建 epoll 实例（只做一次）
epfd = epoll_create()

// 第二步：注册/修改/删除关注的 fd（按需操作）
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &event)   // 加入一个 fd
epoll_ctl(epfd, EPOLL_CTL_MOD, fd, &event)   // 修改关注的事件
epoll_ctl(epfd, EPOLL_CTL_DEL, fd)            // 移除一个 fd

// 第三步：等待事件（只返回就绪的 fd）
n = epoll_wait(epfd, events[], maxevents, timeout)
// events[] 里只有就绪的 fd，不需要遍历所有
```

#### 为什么 epoll 不需要遍历所有 fd？

**关键：内核维护了一棵红黑树 + 一个就绪链表。**

- **红黑树**：存储所有注册的 fd（高效的插入、查找、删除）
- **就绪链表**：当某个 fd 有事件时，内核通过回调机制把它加入就绪链表
- `epoll_wait` 只需要把就绪链表里的 fd 复制给用户态，不需要遍历红黑树

```text
注册 1 万个 fd → 红黑树存着
其中 3 个有数据到来 → 内核回调把这 3 个加入就绪链表
epoll_wait → 只返回这 3 个，O(1)（就绪数少时）
```

#### epoll 的三个 API

| API | 作用 | 调用频率 |
|---|---|---|
| `epoll_create` | 创建 epoll 实例（内核数据结构） | 一次 |
| `epoll_ctl` | 增/删/改关注的 fd | 按需 |
| `epoll_wait` | 等待并返回就绪的 fd | 循环调用 |

#### 对比 select/poll

| | select | poll | epoll |
|---|---|---|---|
| fd 上限 | 1024 | 无硬上限 | 无硬上限 |
| 内核侧数据结构 | 无（每次重建） | 无（每次重建） | 红黑树（持久保存） |
| 如何找就绪 fd | 遍历所有 fd | 遍历所有 fd | 回调 + 就绪链表 |
| 每次调用传递 | 全量 fd | 全量 fd | 只返回就绪 fd |
| 时间复杂度 | O(n) | O(n) | O(1)（就绪数少时） |
| 适用场景 | 少量连接 | 中等连接 | 大量连接、高并发 |

#### 为什么 epoll 适合高并发？

- **不用每次传所有 fd**：fd 注册一次就行，不用反复传
- **不用遍历所有 fd**：内核用回调机制，有事件才通知
- **只返回就绪的**：用户拿到的一定是有数据的，不用挨个检查

> 一句话：**select/poll 每次都要”全班点名”，epoll 是”有事的学生举手”。**

---

### 5. kqueue

**FreeBSD / macOS 专有的 IO 多路复用机制，设计理念和 epoll 类似，但更通用。**

#### 为什么会有 kqueue？

epoll 是 Linux 专有的。BSD 系（FreeBSD、macOS）没有 epoll，它们有自己的机制：**kqueue**。

实际上 kqueue 比 epoll 出现得更早（2000 年 vs Linux 2.5.44，2002 年）。

#### 工作方式

```text
// 第一步：创建 kqueue 实例
kq = kqueue()

// 第二步：注册关注的事件（通过 kevent 结构体）
kevent(kq, changelist, nchanges, NULL, 0, NULL)

// 第三步：等待事件
kevent(kq, NULL, 0, eventlist, nevents, timeout)
```

#### kqueue 和 epoll 的相似点

- 都是”注册一次，反复等待”的模式
- 都在内核维护数据结构，不用每次全量传递 fd
- 都只返回就绪的事件，不需要线性遍历
- 性能都远好于 select/poll

#### kqueue 比 epoll 更灵活的地方

epoll 只能监控**文件描述符**（socket、pipe、文件等）。
kqueue 可以监控的东西更广泛：

| 监控对象 | epoll | kqueue |
|---|---|---|
| socket 可读/可写 | ✅ | ✅ |
| 文件描述符事件 | ✅ | ✅ |
| 进程退出 | ❌ | ✅（`EVFILT_PROC`） |
| 信号 | ❌（需配合 signalfd） | ✅（`EVFILT_SIGNAL`） |
| 文件系统变化（如目录新增文件） | ❌（需 inotify） | ✅（`EVFILT_VNODE`） |
| 定时器 | ❌（需 timerfd） | ✅（`EVFILT_TIMER`） |
| 用户自定义事件 | ❌（需 eventfd） | ✅（`EVFILT_USER`） |

kqueue 通过 `filter`（过滤器）的概念统一了这些不同类型的事件源，而 Linux 则需要为每种需求单独创建一种 fd（signalfd、timerfd、eventfd、inotify……）。

#### epoll 比 kqueue 更灵活的地方

- epoll 的 `EPOLLONESHOT`：一个 fd 事件触发一次后自动禁用，直到用户重新启用（方便控制”同一 fd 只被一个线程处理”）
- epoll 的 `EPOLLEXCLUSIVE`：避免”惊群效应”（多个 epoll 同时监听同一个 fd 时，只唤醒一个）

#### 平台对应关系

| 操作系统 | IO 多路复用机制 |
|---|---|
| Linux | epoll |
| macOS / FreeBSD | kqueue |
| Windows | IOCP（更重，但是真正的异步 IO） |
| 跨平台抽象库 | libuv、libevent、boost.asio（内部自动选 epoll/kqueue/IOCP） |

#### 面试怎么答

“Linux 上高并发用 epoll，macOS/FreeBSD 用 kqueue，两者设计理念类似——都在内核维护数据结构，通过回调通知就绪事件，避免 select/poll 的 O(n) 遍历问题。kqueue 比 epoll 更通用，可以监控文件系统变化、信号、进程等更多事件源。实际工程中一般用 libuv、libevent 等跨平台库，它们会根据平台自动选择底层机制。”

---

### 5.5 IOCP（I/O Completion Ports）

**Windows 专有的高并发 IO 机制，和 epoll/kqueue 有本质区别——它是真正的异步 IO。**

#### 先理解一个关键区别：同步 IO 多路复用 vs 异步 IO

epoll、kqueue、select、poll 都属于**同步 IO 多路复用**：

```text
同步 IO 多路复用（epoll/kqueue）：
  1. 告诉内核”帮我盯这些 fd”（注册）
  2. 等内核告诉你”fd 就绪了”（等待）
  3. 你自己去 read/write（你自己做 IO）

异步 IO（IOCP）：
  1. 你发起 read 请求，附带一个缓冲区
  2. 内核自己去读，读完了通知你
  3. 你拿到的直接就是数据——你从没自己调过 read
```

> **epoll/kqueue** = “内核告诉你什么时候可以读了，你自己去读”
> **IOCP** = “你告诉内核去读，内核读完了把结果给你”

#### IOCP 的工作方式

```text
// 第一步：创建 Completion Port
iocp = CreateIoCompletionPort(INVALID_HANDLE_VALUE, NULL, 0, worker_threads)

// 第二步：把 socket 句柄关联到这个 Port
CreateIoCompletionPort(socket, iocp, completion_key, 0)

// 第三步：发起异步 IO 操作（比如投递一个 Read 请求）
WSARecv(socket, &buffer, ...)   // 立即返回，不阻塞
                                 // 内核在后台读数据

// 第四步：工作线程等完成通知
GetQueuedCompletionStatus(iocp, &bytes_transferred, &completion_key, &overlapped, ...)
// 阻塞等待 → 内核读完后唤醒，直接给你数据和长度
```

#### 核心概念

**Completion Port（完成端口）**：内核维护的一个队列。所有关联到这个端口的异步 IO 操作完成后，结果会被投递到这个队列里。

**Completion Key**：每次关联句柄时附带的自定义标识，用来区分”这个完成通知是哪个连接/请求的”。

**工作线程池**：多个线程同时调用 `GetQueuedCompletionStatus` 等待，内核自动做负载均衡——同一时刻最多让 `worker_threads` 个线程运行，避免线程过多导致上下文切换开销。

#### IOCP 的杀手锏：内核自动管理线程池

```text
假设你设了并发线程数 = 4，但有 8 个工作线程在等：

  内核保证：同一时刻最多只有 4 个线程从 GetQueuedCompletionStatus 返回
  其余线程继续睡眠等待

好处：
  - 不需要自己管理线程池大小
  - 避免所有线程同时被唤醒（惊群效应）
  - 减少上下文切换
```

epoll 要自己管理线程池（比如用 Reactor 模式），IOCP 把线程池管理也内置了。

#### IOCP vs epoll/kqueue：本质区别

| | epoll / kqueue | IOCP |
|---|---|---|
| 模型 | IO 多路复用（同步） | 完成端口（异步） |
| 通知内容 | “fd 就绪了，你可以去读/写了” | “IO 已经完成了，数据在这里” |
| 谁做 read/write | 用户代码自己做 | 内核帮你做 |
| 是否需要非阻塞 fd | 需要（特别是 ET 模式） | 不需要，本身就是异步的 |
| 线程池管理 | 用户自己管理 | 内核自动管理 |
| 复杂度 | 相对简单 | 相对复杂（需要管理 Overlapped 结构） |
| 平台 | Linux / macOS | Windows |

#### 为什么 IOCP 被称为”真正的异步 IO”？

在 IO 理论中，**真正的异步 IO**（Asynchronous IO / AIO）定义是：

1. 你发起 IO 请求，**立即返回**
2. 内核在后台完成**整个 IO 操作**（包括把数据从内核缓冲区复制到用户缓冲区）
3. 完成后通知你

epoll 虽然告诉你”可以读了”，但实际的 `read()` 系统调用还是你自己做的——数据从内核空间复制到用户空间这一步还是同步阻塞的。所以 epoll 属于**同步非阻塞**模型（`recv` 不会阻塞，但数据拷贝仍是同步的）。

IOCP 的 `WSARecv` 则是：你给内核一个缓冲区，内核读完后把数据直接放进去再通知你，你从没体验过”等待数据拷贝”的过程。这才是 POSIX 定义的**异步 IO**。

> 不过，在 Linux 上 epoll + 非阻塞 IO 的性能已经足够好了，绝大多数高并发服务并不需要真正的异步 IO。Linux 也有 `io_uring`（更现代的异步 IO 接口），但那是另一个话题。

#### 面试怎么答

“Windows 上高并发 IO 用 IOCP，它是真正的异步 IO——你发起 IO 请求后内核在后台完成，完成后通知你结果，而 epoll/kqueue 只是告诉你 fd 就绪了，你自己去做 read/write。
IOCP 还内置了线程池管理和负载均衡，避免惊群效应。
实际工程中 libuv 等跨平台库会自动选 epoll/kqueue/IOCP。”

#### 平台对应关系（更新版）

| 操作系统 | 高并发 IO 机制 | 模型 |
|---|---|---|
| Linux | epoll | 同步 IO 多路复用 |
| macOS / FreeBSD | kqueue | 同步 IO 多路复用 |
| Windows | IOCP | 异步 IO（完成端口） |
| Linux（更新） | io_uring | 异步 IO（更现代） |
| 跨平台抽象库 | libuv、libevent、boost.asio | 自动选择底层机制 |

### 6. LT 和 ET（epoll 的两种触发模式）

#### LT（Level Triggered，水平触发，epoll 默认模式）

> **只要 fd 上还有数据没读完，每次 `epoll_wait` 都会告诉你这个 fd 就绪。**

```text
socket 收到 1000 字节
  epoll_wait → 返回”socket 可读”
  你读了 400 字节，还剩 600 字节
  下次 epoll_wait → 还是返回”socket 可读”（因为还有数据）
  ...
  直到读完为止
```

特点：
- 简单安全，不容易漏事件
- 不用担心”没读完下次就不通知了”
- 但可能多次通知同一个 fd

#### ET（Edge Triggered，边缘触发）

> **只在 fd 状态发生变化时通知一次**（从无数据 → 有数据），不管你读没读完。

```text
socket 收到 1000 字节
  epoll_wait → 返回”socket 可读”（通知一次）
  你读了 400 字节，还剩 600 字节
  下次 epoll_wait → 不再通知！（因为没有新数据到来）
  除非对方又发了新数据，才会再次通知
```

特点：
- 更高效，减少通知次数
- 但要求你**一次性读完/写完**所有数据（用循环 read 直到 `EAGAIN`）
- 如果没读完就不管了，数据就”悬在那”直到下次新数据到来

#### LT vs ET

| | LT（水平触发） | ET（边缘触发） |
|---|---|---|
| 通知时机 | 有数据就通知 | 状态变化时通知一次 |
| 没读完会怎样 | 下次继续通知 | 不通知，直到新数据到来 |
| 编程难度 | 简单 | 需要循环 read/write 直到 EAGAIN |
| 性能 | 略低（可能多次通知） | 略高（通知次数少） |
| 常见搭配 | 阻塞或非阻塞 fd 都行 | 必须用非阻塞 fd |

> 面试建议：除非你清楚知道为什么需要 ET，否则用 LT（默认）就对了。Nginx 用的是 ET。

---

### 7. Reactor 模型

Reactor 的直觉是：

- 事件循环等事件
- 事件来了分发给对应逻辑

#### 单线程 Reactor

```text
一个线程：
  while true:
      events = epoll_wait(...)
      for event in events:
          if 是新连接:
              accept()
          elif 是可读:
              read() → 解析 → 处理 → write()
```

简单，但单线程处理能力有限。

#### 多线程 Reactor（常见变体）

```text
主线程（Acceptor）：
  epoll 监听 listenfd
  新连接来了 → accept → 把 connfd 分发给工作线程

工作线程（Worker）：
  各自 epoll 监听自己负责的 connfd
  读写 + 业务处理
```

#### 面试一句话

> Reactor = “事件循环 + 回调分发”。epoll 负责等事件，事件来了分给对应的处理逻辑。
> 单线程 Reactor 适合 IO 密集轻逻辑，多线程 Reactor 适合需要并行处理的场景。

很多高并发网关、聊天室、游戏接入层都会用类似思路。

## 十、面向游戏后端的并发设计

> 这一章把前面的 OS 和并发知识落地到游戏服务端的真实场景。

### 1. 游戏后端的架构长什么样

先建立一个整体图景。一个典型的游戏服务端（如 MOBA、吃鸡、MMO）通常不是单体服务，而是**一组分工明确的服务进程**：

```text
客户端
  │
  ▼
接入层（Gateway / Relay）
  │
  ├──▶ 登录服（Login）
  ├──▶ 匹配服（Matchmaking）
  ├──▶ 房间服 / 战斗服（Room / Battle）
  ├──▶ 聊天服（Chat）
  ├──▶ 公会服（Guild）
  ├──▶ 排行榜服（Ranking）
  ├──▶ 背包 / 道具服（Inventory）
  └──▶ 数据库代理（DB Proxy）
```

每个服务内部都有自己的并发模型。不同服务的特点不同，设计思路也不同。

### 2. 为什么游戏服务不喜欢”大共享 + 大锁”

假设你用一个全局的大 map 存所有玩家的状态，所有线程共享、加一把大锁：

```text
全局玩家表 + 一把大 mutex

线程 1：处理玩家 A 的移动 → 加锁 → 改位置 → 解锁
线程 2：处理玩家 B 的攻击 → 加锁 → 改血量 → 解锁
线程 3：处理玩家 A 的使用道具 → 加锁 → 改背包 → 解锁
线程 4：排行榜更新 → 加锁 → 读所有玩家 → 解锁
```

问题：

- **锁竞争严重**：所有线程抢同一把锁，变成串行执行
- **尾延迟很高**：某个线程持锁时间稍长（比如排行榜遍历），其他线程全在等
- **热点放大**：一个热门玩家（比如大主播）的所有操作都走同一把锁，可能拖垮整个服务
- **连锁反应**：一个慢操作（数据库查询、磁盘 IO）如果放在锁里面，整个系统都会卡住

> 本质问题：**不相关的事情被同一把锁耦合在一起了。** 玩家 A 的移动和玩家 B 的攻击本来互不干扰，却要排队等同一把锁。

### 3. 房间服 / 战斗服常见思路

#### 核心原则：房间内串行，房间间并行

```text
线程 1：负责房间 101（单线程事件循环）
  - 处理玩家 A 的移动
  - 处理玩家 B 的攻击
  - 处理技能结算
  - 处理 tick 更新
  所有这些操作串行执行，不需要锁

线程 2：负责房间 102（单线程事件循环）
  - 同上，独立运行

线程 3：负责房间 103、104（如果房间轻量，一个线程可以管多个）
  ...
```

为什么这样做？

#### 房间内为什么可以串行？

游戏战斗的逻辑本质上是**帧驱动的**：

```text
每一帧（比如 20ms）：
  1. 收集本帧所有玩家的输入
  2. 按确定性的顺序处理
  3. 更新状态（位置、血量、技能 CD、buff……）
  4. 广播状态给所有玩家
```

这些操作**天然就是串行的**——你很难把”玩家 A 攻击玩家 B”和”玩家 B 使用道具”并行处理，因为它们之间有因果关系（攻击可能打死 B，B 就不能用道具了）。

既然逻辑上就必须串行，那就不需要锁，用一个线程串行处理最简单、最高效。

#### 房间间为什么可以并行？

房间 101 里发生的事情和房间 102 **完全无关**。它们不共享状态（或者只共享少量只读数据，比如配置表）。所以自然可以并行。

#### 好处

- **零锁**：房间内不需要任何锁，逻辑简单
- **状态一致性天然保证**：串行执行 = 天然有序，不存在竞态条件
- **性能可预测**：每个房间的处理时间独立，不会互相影响
- **故障隔离**：一个房间的 bug 不会直接影响其他房间

### 4. 按玩家、房间、分区做串行化

“局部串行、整体并行”这个思路可以灵活运用在不同粒度上：

#### 粒度选择

| 粒度 | 串行范围 | 并行范围 | 典型场景 |
|---|---|---|---|
| 按玩家 | 同一玩家的所有请求串行 | 不同玩家并行 | 背包操作、个人任务 |
| 按房间 | 同一房间的所有操作串行 | 不同房间并行 | 战斗、匹配房间 |
| 按分区 / 分线 | 同一区的操作串行 | 不同区并行 | MMO 的分线地图 |
| 按公会 | 同一公会的操作串行 | 不同公会并行 | 公会战、公会管理 |

#### 实现方式：哈希分片

```text
// 把实体（玩家/房间/公会）映射到固定的处理线程
thread_id = hash(entity_id) % num_threads

// 每个线程维护自己的事件队列
// 所有发给这个实体的消息都投递到对应线程的队列里
// 线程串行消费自己队列里的消息
```

这样：
- 同一实体的消息一定会落到同一个线程，天然串行
- 不同实体的消息会分散到不同线程，天然并行
- 不需要锁

### 5. actor / mailbox 思想

这是”局部串行、整体并行”的形式化表达。

#### 什么是 actor 模型

actor 是一种并发编程范式，核心规则：

1. **每个 actor 有自己独占的状态**，外界不能直接访问
2. **actor 之间只能通过异步消息通信**
3. **每个 actor 串行处理自己的消息队列**
4. **收到消息后可以：创建新 actor、发消息给其他 actor、改变自己的状态**

```text
玩家 actor A              房间 actor R              玩家 actor B
  │                         │                         │
  │─── “我要移动到(x,y)” ──▶│                         │
  │                         │ 串行处理：               │
  │                         │   校验移动合法性          │
  │                         │   更新 A 的位置          │
  │◀── “移动成功” ──────────│─── “广播 A 移动了” ────▶│
  │                         │                         │
```

#### 为什么游戏后端天然适合 actor

游戏里的几乎所有实体都天然有”独占状态 + 消息驱动”的特征：

| Actor 类型 | 独占状态 | 接收的消息 |
|---|---|---|
| 玩家 actor | 位置、血量、背包、任务进度 | 移动、使用道具、接任务、被攻击 |
| 房间 actor | 房间内所有玩家状态、游戏进度 | 玩家加入/离开、技能释放、tick |
| 公会 actor | 成员列表、公会资金、权限 | 加入/退出、捐献、升级 |
| 匹配 actor | 匹配队列 | 加入匹配、取消匹配、匹配成功 |

#### actor 的关键好处

- **无锁**：状态独占，不需要 mutex
- **无死锁**：没有共享锁，就没有循环等待
- **天然分布式**：actor 可以在不同机器上，只需要消息能送达就行
- **易于推理**：每个 actor 的逻辑是单线程的，理解和调试都更简单

#### 实际实现

语言/框架层面：
- **Erlang/Elixir**：天生的 actor 语言，BEAM VM 的每个”进程”就是 actor
- **Akka（Scala/Java）**：JVM 上最成熟的 actor 框架
- **Go**：goroutine + channel 类似 actor，但更轻量
- **C++ 自研**：很多游戏公司自研的框架底层就是这个思路（消息队列 + 单线程消费）

### 6. 游戏后端的线程模型

综合前面的知识，一个典型的游戏服务端线程模型：

```text
┌─────────────────────────────────────────────┐
│                 服务进程                       │
│                                              │
│  ┌──────────┐  网络线程（1~N 个）              │
│  │ Net IO   │  epoll/kqueue + Reactor         │
│  │ Thread   │  负责：收包、解码、分发            │
│  └────┬─────┘                                │
│       │ 解码后的消息                            │
│       ▼                                      │
│  ┌──────────┐  逻辑线程（N 个，按房间/玩家分片）   │
│  │ Logic    │  各自有独立的消息队列               │
│  │ Thread 1 │  串行处理队列中的消息               │
│  │──────────│                                │
│  │ Logic    │  不需要锁                        │
│  │ Thread 2 │                                │
│  │──────────│                                │
│  │ Logic    │                                │
│  │ Thread 3 │                                │
│  └────┬─────┘                                │
│       │ 需要持久化的数据                          │
│       ▼                                      │
│  ┌──────────┐  IO 线程（1~N 个）                │
│  │ IO       │  异步写数据库、RPC 调用             │
│  │ Thread   │  结果通过回调/消息返回逻辑线程        │
│  └──────────┘                                │
│                                              │
│  ┌──────────┐  定时器线程                       │
│  │ Timer    │  管理 tick、buff 过期、技能 CD 等   │
│  │ Thread   │  到时触发消息给逻辑线程              │
│  └──────────┘                                │
└─────────────────────────────────────────────┘
```

#### 各线程的职责

**网络线程**：
- 用 epoll/kqueue 等待网络事件
- 收到数据后解码协议
- 根据 entity_id 投递到对应逻辑线程的队列
- **不做业务逻辑**，只做 IO 和分发

**逻辑线程**：
- 每个线程负责一批实体（房间/玩家）
- 串行消费消息队列
- 执行游戏逻辑（技能计算、状态更新、AI……）
- 需要持久化时，投递给 IO 线程，不等结果（异步）

**IO 线程**：
- 处理数据库读写、RPC 调用
- 结果通过消息返回给逻辑线程
- 逻辑线程永远不直接阻塞在 IO 上

**定时器线程**：
- 管理所有定时事件
- 到时间后投递消息给逻辑线程

#### 为什么逻辑线程不能阻塞在 IO 上？

```text
错误做法（逻辑线程直接查数据库）：
  逻辑线程处理到”玩家使用道具”
    → 加锁 → 查数据库确认道具存在 → 等 5ms → 改状态 → 解锁
    这 5ms 里，这个线程负责的所有房间/玩家都在等

正确做法（异步）：
  逻辑线程处理到”玩家使用道具”
    → 投递”查询道具”请求给 IO 线程 → 继续处理下一条消息
    IO 线程查完数据库 → 投递”道具查询结果”给逻辑线程
    逻辑线程下次收到结果消息时再继续处理
```

### 7. 如何减少锁竞争

当某些场景确实需要共享状态时，如何减少锁的开销：

**1. 缩小锁粒度**

```text
// 差：一把大锁锁整个玩家表
lock(&global_mutex)
find(player_id)
modify(player)
unlock(&global_mutex)

// 好：每个玩家一把锁
lock(&player_list[player_id].mutex)
modify(player)
unlock(&player_list[player_id].mutex)
```

**2. 减少共享状态**

- 能用局部变量的就不要用全局变量
- 能用 thread-local 的就不要用共享的
- 配置数据用只读共享（不需要锁），运行时状态尽量本地化

**3. 用分片代替全局锁**

```text
// 把玩家按 ID 分片，每个分片有自己的锁
shard_id = player_id % num_shards
lock(&shard[shard_id].mutex)
// 只锁这一个分片里的玩家
```

**4. 用消息传递代替共享内存修改**

这就是 actor 思路——不直接改别人的状态，而是发消息让它自己改。

**5. 把热点数据本地化**

- 每个逻辑线程缓存自己负责的实体的数据
- 只有跨线程交互时才需要同步
- 类似 CPU Cache 的思路——局部性越好，性能越高

**6. 读写分离**

- 配置表、地图数据等只读数据，全局共享不需要锁
- 如果有”偶尔写”的场景，用 copy-on-write 或读写锁

### 8. 面试怎么答

> “游戏后端的并发设计核心原则是**局部串行、整体并行**。
> 房间/战斗的逻辑天然需要串行处理，所以每个房间分配给一个线程串行执行，不需要锁；
> 不同房间之间互不干扰，天然可以并行。
> 实现上通常用 actor/mailbox 模型——每个实体有独占状态和消息队列，串行消费消息，线程间通过异步消息通信。
> 网络线程用 epoll 处理 IO 和分发，逻辑线程做业务，IO 线程处理数据库和 RPC，定时器线程管理超时。
> 逻辑线程绝不阻塞在 IO 上，避免一个慢操作拖住整个房间。”

## 十一、游戏后端常见并发工程问题

### 1. 顶号和会话并发

同一账号两个连接同时操作时，要避免：

- 双在线
- 旧连接回写新状态
- 重复发消息

### 2. 重连与旧连接并发写状态

断线重连后，旧连接上的延迟包、补发包、回调任务可能晚到。
如果不做 session 校验，很容易把新状态覆盖掉。

### 3. 发奖和扣道具幂等

并发重试、网络超时、重复请求都可能导致：

- 奖励发两次
- 道具扣两次

所以这类操作必须：

- 幂等
- 去重
- 带请求 id

### 4. 热点房间 / 热点公会 / 热点排行榜

这些都容易形成：

- 热锁
- 热 key
- 热 actor

### 5. 慢任务阻塞事件循环

如果你把：

- 慢 SQL
- 慢 RPC
- 重 CPU 计算

塞进单线程事件循环或房间主线程里，整个房间 / 分片都会抖。

### 6. 锁争用导致尾延迟飙升

平均耗时可能还行，但 P99/P999 会明显恶化。
这是很多线上问题最恶心的地方。

## 十二、游戏大厂面试可能问的重点题

### 1. 线程和进程有什么区别

关键答法：

- 进程是资源分配单位
- 线程是调度单位
- 线程共享地址空间但同步更麻烦

### 2. mutex 和 spinlock 区别

- mutex 抢不到通常阻塞，让出 CPU
- spinlock 抢不到通常自旋，适合极短临界区

### 3. CAS 和 mutex 怎么选

- 低冲突、简单原子更新：CAS 可能好用
- 临界区复杂、需要阻塞等待：mutex 更稳

### 4. 读写锁是不是一定更快

不一定。
只有读多写少、临界区足够长时更容易体现价值。

### 5. epoll 为什么更适合高并发

因为它更适合大量 fd，不需要像 select 一样每次全量线性扫描。

### 6. 为什么线程池大小不能乱设

线程太少：

- 吞吐不够

线程太多：

- 切换多
- 内存大
- Cache 被打乱

### 7. 房间服如何做少锁甚至无锁

成熟回答思路：

- 房间内状态单线程推进
- 跨房间并行
- 房间外通过消息投递
- 不共享可变状态，尽量少直接加锁

### 8. 为什么游戏后端很看重幂等

因为：

- 网络会重试
- 重连会补发
- 客户端可能重复请求

如果奖励、扣道具、结算不幂等，就会出严重业务错误。
