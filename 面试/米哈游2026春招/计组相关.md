## 二、为什么要学计组

### 1. 计组到底研究什么

计算机组成原理研究的是：

> 一台计算机内部的各个硬件部件是如何组织起来，并共同执行程序的。

它重点关心的不是“软件逻辑怎么写”，而是：

- 指令怎么表示
- 数据怎么存
- CPU 怎么执行
- 存储系统为什么分层
- 性能瓶颈为什么会出现

### 2. 它和哪些东西强相关

#### 和操作系统的关系

操作系统管理内存、进程、中断、I/O，但这些能力都建立在底层硬件机制上。

例如：

- 页表、TLB、缺页异常
- 中断和异常
- DMA

#### 和编译原理的关系

编译器最终要把高级语言翻译成机器指令。
如果你不知道指令和寄存器是怎么工作的，就很难真正理解代码生成和优化。

#### 和性能优化的关系

很多性能问题表面上是“代码慢”，本质上是：

- Cache miss 太多
- 分支预测失败
- 指令级并行没利用起来
- 内存访问比计算慢很多

### 3. 初学者最容易卡住的点

1. 一上来就背寄存器名字，没先建立全局图
2. 把教材里的术语当成孤立概念
3. 只会背定义，不知道它解决什么问题

这份文档会优先避免这三个坑。

### 4. 这一章你最后要记住什么

计组最核心的一句话其实是：

> 程序 = 指令 + 数据；CPU 负责执行指令，存储系统负责及时提供数据，而性能矛盾的核心常常是“CPU 很快，内存很慢”。

后面所有章节几乎都在围绕这句话展开。

## 三、先建立全局图：一台计算机是怎么工作的

### 1. 程序到底是什么

程序本质上可以看成两类东西：

1. **指令**
2. **数据**

指令告诉 CPU：

- 做什么操作
- 操作哪些数据

数据则存在寄存器、内存、Cache、磁盘等地方。

### 2. 一台计算机最核心的几个部件

你先记住这几个名字：

- **CPU**
- **寄存器**
- **Cache**
- **主存**
- **I/O 设备**
- **总线**

它们的关系可以粗略理解为：

- CPU 负责执行指令
- 寄存器是 CPU 身边最快的小存储
- Cache 是 CPU 和主存之间的高速缓冲
- 主存存放当前运行程序的大部分代码和数据
- 磁盘/SSD 存放长期数据
- I/O 设备负责和外界交互

### 3. 一条指令的大致生命周期

假设程序里有一句：

```c
c = a + b;
```

从硬件角度看，大致是：

1. CPU 先根据 `PC` 去内存取指令
2. 把取回来的指令放到 `IR`
3. 控制器译码，知道这是一条加法相关指令
4. 找到操作数 `a` 和 `b`
5. 把它们送到 ALU
6. ALU 做加法
7. 结果写回寄存器或内存
8. `PC` 指向下一条指令

这就是“取指 -> 译码 -> 执行 -> 写回”的核心直觉。

### 4. 真正拖慢计算机的常常不是算术，而是访存

初学者很容易误以为：

> CPU 的主要工作就是算加减乘除。

但现实里，CPU 常常不是被“计算”拖慢，而是被“等数据”拖慢。

例如：

- 数据不在寄存器里
- 不在 L1 Cache 里
- 不在 L2/L3 里
- 需要去主存拿

而主存相对 CPU 慢得多。
这也是为什么：

- Cache 非常重要
- 局部性原理非常重要
- 程序访问模式非常重要

### 5. 这张全局图要一直带着

后面你学任何一个点，都最好问自己：

- 它在整个系统里解决什么问题？
- 它是为了减少什么开销？
- 它和 CPU 执行一条指令有什么关系？

## 四、数据的表示：计算机到底怎么看数字

这一章解决的是：

> 人类说的“整数、小数、正负数”，机器到底如何存？

### 1. 进制

计算机底层只认二进制。
所以你至少要熟悉：

- 二进制
- 十进制
- 十六进制

十六进制常用是因为：

- 4 个二进制位刚好对应 1 个十六进制位
- 写起来更短

### 2. 原码、反码、补码

这是面试和基础题高频内容。

#### 原码

- 最高位表示符号
- 其余位表示数值

例如在 8 位表示里：

- `+5`：`00000101`
- `-5`：`10000101`

#### 反码

- 正数反码和原码相同
- 负数反码：符号位不变，其余位按位取反

#### 补码

- 正数补码和原码相同
- 负数补码：反码 + 1

### 3. 为什么机器更喜欢补码

补码的核心优点是：

1. **加法和减法可以统一处理**
2. **0 的表示唯一**
3. **硬件实现更简单**

比如：

- 原码和反码都存在 `+0` 和 `-0`
- 补码只有一个 0

所以现代计算机基本都使用补码表示有符号整数。

#### 一个很常见的追问

如果是 8 位补码，可表示范围是：

- `-128` 到 `127`

负数会比正数多一个，这也是为什么 `-128` 的绝对值没法在同样的 8 位有符号补码里表示出来。

### 4. 移码

移码常用于：

- 表示浮点数的指数部分

它的思路是给真值整体加一个偏移量，这样比较大小更方便。

### 5. 定点数和浮点数

#### 定点数

小数点位置默认固定。
适合范围固定、硬件简单的场景。

#### 浮点数

小数点位置可以“浮动”。
这样可以表示更大范围的实数。

### 6. IEEE 754 浮点数直觉

浮点数通常由三部分组成：

1. 符号位
2. 指数
3. 尾数

它的核心思想类似科学计数法：

> 一个数 = 尾数 × 基数的指数次方

#### 你至少要知道的点

- 浮点数能表示的范围很大
- 但很多十进制小数无法被精确表示
- 所以会出现精度误差

例如经典问题：

```text
0.1 + 0.2 != 0.3
```

本质不是语言 bug，而是二进制浮点表示精度问题。

#### 高频误区

- 浮点数不是“精确存十进制小数”
- 浮点数比较通常不应直接用 `\==` 判断业务相等，而更常比较误差范围

### 7. 大小端

大小端说的是：

> 多字节数据在内存里按什么顺序存放。

#### 大端

高位字节放低地址。

#### 小端

低位字节放低地址。

现代 x86 通常是小端。

### 8. 数据对齐

数据对齐说的是：

> 某些类型的数据放在特定倍数的地址上，CPU 读写会更高效。

为什么要对齐：

- 硬件访问更自然
- 可能减少访存次数
- 简化总线传输和实现

结构体对齐会带来一个常见现象：

> 字段字节数加起来看起来是 13 字节，实际结构体大小可能是 16 字节甚至更大。

这是编译器插入 padding 的结果，本质上是在用少量空间换更自然、更高效的访问。

#### 高频误区

- 对齐不是“语法要求”，本质是性能和硬件访问习惯
- 补码是重点，反码和原码主要是为了理解补码为什么出现

## 五、存储系统：为什么寄存器最快、磁盘最慢

### 1. 存储层次结构

计算机不会只用一种存储器，而是分层：

1. 寄存器
2. L1 / L2 / L3 Cache
3. 主存
4. SSD / HDD

从上到下通常是：

- 越往上越快
- 越往上越贵
- 越往上容量越小

### 2. 为什么不能只用最快的存储

因为最快的存储通常：

- 容量小
- 价格高
- 功耗高

如果只用寄存器或超大 Cache 来替代主存，成本会极不现实。

所以现实系统采用的是：

> 小而快的上层 + 大而慢的下层

### 3. 每一层大概负责什么

#### 寄存器

- 离 CPU 最近
- 速度最快
- 容量最小
- 存 CPU 当前最直接要用的数据

#### Cache

- 缓冲主存中的热点数据
- 让 CPU 少等主存

#### 主存

- 存当前运行程序的主要代码和数据

#### 外存

- 持久化存储
- 容量大，但远慢于主存

### 4. 局部性原理

局部性原理是整个存储层次设计的基础。

#### 时间局部性

如果一个数据刚被访问过，接下来很可能还会再访问。

#### 空间局部性

如果一个数据被访问了，附近的数据也很可能很快被访问。

### 5. 为什么局部性这么重要

因为程序经常表现出这些行为：

- 循环里反复使用同一批变量
- 顺序遍历数组
- 指令通常也是连续执行的

这就让 Cache 很有价值。

### 6. 学计组时要形成的存储直觉

很多性能问题，本质上不是“算法算得慢”，而是：

- 数据拿得慢
- 拿数据的方式不友好

## 六、Cache：计组面试最核心的存储问题

### 1. Cache 为什么存在

CPU 很快，主存相对很慢。
如果 CPU 每次都直接等主存，性能会被严重拖垮。

所以引入 Cache：

> 把最近常用的数据放在更快的小存储里。

### 2. Cache 的几个核心词

#### 命中 Hit

CPU 要的数据就在 Cache 里。

#### 缺失 Miss

CPU 要的数据不在 Cache 里，需要去更下层拿。

#### 命中率

命中的次数 / 总访问次数。

### 3. Cache 行/块

Cache 不是按“一个字节一个字节”调数据，而通常按一个块来搬。

这个块常叫：

- Cache block
- Cache line

这样做是因为空间局部性。

### 4. 三种典型映射方式

#### 直接映射

每个主存块只能映射到 Cache 的唯一位置。

优点：

- 简单
- 快

缺点：

- 容易冲突 miss

#### 全相联

主存块可以放到 Cache 任何位置。

优点：

- 冲突少

缺点：

- 硬件复杂
- 查找成本高

#### 组相联

折中方案。
先分组，主存块只能映射到某一组内，但组内可选多个位置。

工程里很常见。

### 5. Cache miss 的几类直觉

常见可以这样理解：

1. **冷启动 miss**：第一次访问，Cache 里本来就没有
2. **容量 miss**：Cache 太小，装不下工作集
3. **冲突 miss**：映射方式导致热点块互相顶掉

### 6. 替换策略

当一组放满后，新的块来了，要替换谁？

#### LRU

最近最少使用。

#### FIFO

先进先出。

#### Random

随机替换。

工程里常用近似 LRU，而不一定是真正的精确 LRU。

### 7. 写策略

#### 写直达 Write Through

写 Cache 的同时也写主存。

优点：

- 一致性更直接

缺点：

- 写开销大

#### 写回 Write Back

先只写 Cache，等块被替换时再写回主存。

优点：

- 写性能通常更好

缺点：

- 控制更复杂

### 8. 写分配和非写分配

#### 写分配

写 miss 时，先把块调入 Cache，再写。

#### 非写分配

写 miss 时，不把块调入 Cache，直接写下层。

### 9. 为什么 Cache miss 很贵

因为 miss 之后，CPU 可能要：

1. 发起更下层访问
2. 等待数据返回
3. 把整块搬入 Cache
4. 可能还触发替换和写回

如果一路 miss 到主存，代价就会明显放大。

### 10. 伪共享的直觉

伪共享通常出现在多核环境。

现象是：

- 两个线程改的是不同变量
- 但这两个变量恰好落在同一个 Cache line
- 结果两个核心不断让这个 Cache line 失效、同步

虽然逻辑上“没共享同一个变量”，但硬件上共享了同一个缓存块，所以性能会变差。

### 11. 写程序时哪些行为更友好

- 顺序访问数组通常比随机访问快
- 按行访问二维数组通常比按列访问更友好
- 频繁跳来跳去的数据结构更不友好

### 12. 一个必须会的公式

平均访存时间常写成：

> `AMAT = 命中时间 + Miss率 × Miss代价`

这个公式很重要，因为它说明：

- 命中率只要下降一点，如果 miss 代价很高，总体性能就会明显变差
- 所以很多性能优化，本质上是在降低 miss 率或 miss 代价

## 七、主存与虚拟存储的连接点

这一章虽然更像计组和操作系统交界处，但面试很爱混着问。

### 1. 页和页表

虚拟内存把地址空间按固定大小分成页。

- 虚拟地址空间中的页：虚页
- 物理内存中的页框：页框

页表记录：

> 虚页号对应哪个物理页框

### 2. 为什么要有虚拟地址

虚拟地址带来几个好处：

1. 每个进程看到独立地址空间
2. 便于内存管理
3. 便于保护与隔离
4. 让程序不必关心真实物理内存布局

### 3. TLB 是什么

TLB 可以理解成：

> 页表项的高速缓存

因为每次访存前，如果都要先查一遍页表，再去查真正数据，开销太大。

所以把最近常用的地址转换结果缓存起来。

### 4. 缺页异常的直觉

如果访问的页当前不在物理内存里，就会触发缺页异常。

操作系统会：

1. 找到对应页在磁盘上的位置
2. 把它换入主存
3. 更新页表
4. 再重新执行这次访问

### 5. TLB、页表、Cache 三者不要混

它们缓存的是不同东西：

- **TLB**：缓存“地址转换结果”
- **Cache**：缓存“真正的数据/指令内容”
- **页表**：提供完整的地址映射关系

### 6. 为什么这块必须懂

因为现实中的访存不是简单的“CPU 直接用地址拿数据”，而通常是：

1. 虚拟地址先查 TLB / 页表
2. 转成物理地址
3. 再去 Cache / 主存取数据

#### 一组非常容易混的概念

- **TLB miss**：只是地址翻译缓存没命中，不代表页不在内存里
- **缺页**：目标页根本不在物理内存里，需要操作系统介入，从外存调页

所以：

> `TLB miss != 缺页`

## 八、指令系统：CPU 到底执行的是什么

### 1. 什么是指令

指令就是 CPU 能理解的最基本操作命令。

例如：

- 加
- 减
- 读内存
- 写内存
- 跳转

### 2. 指令一般由什么组成

通常可以理解为两部分：

1. **操作码**
2. **地址码 / 操作数信息**

操作码决定“做什么”，地址码决定“操作谁”。

从更完整的视角看，指令格式是在平衡：

- 信息表达能力
- 指令长度
- 硬件译码复杂度

教材里常见的零地址、一地址、二地址、三地址指令，本质上就是不同表达能力和编码成本之间的取舍。

### 3. 常见寻址方式

你不一定要死抠每种编码细节，但要理解为什么要有不同寻址方式。

#### 立即寻址

操作数直接写在指令里。

#### 直接寻址

指令里给出操作数地址。

#### 寄存器寻址

操作数在寄存器里。

#### 寄存器间接寻址

寄存器里存的是地址。

#### 基址 / 变址寻址

常用于数组、结构体、栈帧等复杂地址计算。

#### 相对寻址

常写成 `PC + 偏移量`。
它特别适合分支跳转和位置无关代码。

### 4. CISC 和 RISC

#### CISC

复杂指令集。

特点：

- 指令种类多
- 单条指令功能可能很复杂

#### RISC

精简指令集。

特点：

- 指令更规整
- 更适合流水线
- 常强调 load/store 分离

#### 常见理解

- x86 常被视为 CISC
- ARM 常被视为 RISC

#### 一个成熟一点的理解

不要把 CISC 和 RISC 理解成“一个先进一个落后”，更准确地说，它们是不同设计哲学。现代处理器内部实现也常常比教科书标签更复杂，例如复杂指令可能在内部再拆成更简单的微操作。

### 5. 为什么指令系统设计重要

因为它会直接影响：

- 硬件实现复杂度
- 流水线友好程度
- 编译器优化方式

---

## 九、CPU 的核心组成

### 1. CPU 可以粗略分成两部分

1. **运算器**
2. **控制器**

### 2. 运算器做什么

运算器负责：

- 算术运算
- 逻辑运算
- 参与数据搬运

核心部件是 **ALU**。

### 3. 控制器做什么

控制器负责：

- 取指
- 译码
- 产生控制信号
- 组织各部件按顺序协作

### 4. 高频寄存器

#### PC

Program Counter，程序计数器。
存下一条要执行的指令地址。

#### IR

Instruction Register，指令寄存器。
存当前正在分析/执行的指令。

#### MAR

Memory Address Register，存储器地址寄存器。
给出要访问的内存地址。

#### MDR

Memory Data Register，存储器数据寄存器。
存放从内存读出的数据或准备写入内存的数据。

#### PSW

程序状态字。
常包含条件码、标志位、中断允许等状态信息。

### 5. 数据通路直觉

数据通路本质上就是：

> 数据在寄存器、ALU、总线、内存之间流动的路线。

理解 CPU，不是背零件名，而是理解数据怎么流。

访存时常见的一个直觉过程是：

1. 地址先进入 `MAR`
2. 内存返回的数据先进入 `MDR`
3. 再由控制器决定送给 `IR`、寄存器或 ALU

这也是为什么 `MAR` 和 `MDR` 会被单独强调。

---

## 十、指令执行流程

### 1. 最经典的五步直觉

1. 取指
2. 译码
3. 执行
4. 访存
5. 写回

并不是每条指令都一定完整经过五步里的每一步，但这套框架非常重要。

### 2. 取指

CPU 根据 `PC` 去取下一条指令。

常见过程：

1. `PC -> MAR`
2. 发起内存读
3. 指令进入 `MDR`
4. 再送入 `IR`
5. `PC` 更新

### 3. 译码

控制器分析 `IR` 中的操作码和操作数信息，决定：

- 用哪些寄存器
- 是否访问内存
- ALU 做什么运算

### 4. 执行

ALU 或其他功能单元执行实际操作：

- 加减乘除
- 地址计算
- 比较
- 跳转判断

### 5. 访存

如果这条指令需要访问内存，比如 load/store，就要执行访存阶段。

### 6. 写回

把结果写回寄存器，或者在 store 场景写入内存。

### 7. 跳转指令为什么特别

因为它会改变 `PC`，从而影响后续取指。
这也是后面流水线里控制冒险的来源。

### 8. 中断会在什么地方打断

通常不是在指令执行到一半随便切走，而是更常在某条指令完成之后，再进入中断响应流程。

这个说法在学习阶段非常重要，因为它保证机器状态更容易保持一致，也更便于理解“保存现场、处理中断、恢复现场”的流程。

---

## 十一、控制器与控制方式

### 1. 控制器到底在控制什么

控制器通过发出各种控制信号，让各个部件按时序协同。

例如：

- 哪个寄存器该读
- 哪个寄存器该写
- ALU 执行什么操作
- 是否访问内存

### 2. 硬布线控制

用固定逻辑电路直接产生控制信号。

优点：

- 快

缺点：

- 修改和扩展麻烦

### 3. 微程序控制

把复杂控制过程拆成更细粒度的微操作，用微指令组织。

优点：

- 灵活
- 设计和扩展更方便

缺点：

- 通常比纯硬布线更慢一些

### 4. 直觉理解

- 硬布线像“写死的组合逻辑”
- 微程序像“硬件层的小解释器”

#### 高频误区

- 微程序不是普通意义上的应用软件
- 一条机器指令不等于一条微指令

机器指令面向程序员或编译器，微指令面向控制器内部实现。

---

## 十二、单周期、多周期、流水线 CPU

### 1. 单周期 CPU

一条指令在一个时钟周期内完成全部步骤。

优点：

- 直观
- 简单易理解

缺点：

- 时钟周期必须迁就最慢指令
- 资源利用率低

### 2. 多周期 CPU

把一条指令拆成多个阶段，每个阶段占一个或多个周期。

优点：

- 不同指令可用不同周期数
- 时钟周期可以更短

缺点：

- 控制更复杂

### 3. 流水线 CPU

把多条指令的不同阶段重叠执行。

类比工厂流水线：

- 一条指令在译码时
- 下一条可能已在取指
- 上一条可能已在执行

### 4. 三者核心差异

- 单周期：简单，但浪费
- 多周期：更灵活
- 流水线：提高吞吐

### 5. 一个关键点

流水线提升的是：

> 单位时间内完成的指令数

但不代表单条指令一定变快。

这也是“吞吐率”和“单条指令延迟”必须区分开的原因。

---

## 十三、流水线：计组面试第二大重点

### 1. 为什么需要流水线

如果每条指令都必须完整执行完，下一条才能开始，很多硬件资源会闲着。

流水线的目标是：

> 让不同指令在不同阶段并行重叠，提高吞吐率。

### 2. 吞吐率和加速比

#### 吞吐率

单位时间完成多少条指令。

#### 加速比

流水线前后性能提升倍数。

理想情况下，`k` 段流水线装满后可以接近做到“每拍完成 1 条指令”，理论加速比接近 `k`。但现实里会被阶段不均衡、流水寄存器开销、冒险和 cache miss 明显拉低。

### 3. 三类冒险

#### 结构冒险

硬件资源冲突。
例如同一时刻两条指令都想用同一部件。

#### 数据冒险

后一条指令依赖前一条还没写回的结果。

#### 控制冒险

分支跳转导致后续该取哪条指令不确定。

### 4. 数据冒险为什么常见

例如：

```text
add r1, r2, r3
sub r4, r1, r5
```

第二条指令需要 `r1`，但第一条可能还没把结果真正写回。

### 5. 前递 / 转发

如果结果已经算出来，但还没正式写回寄存器，可以直接把结果从流水线后面的阶段“转发”给前面的需要者。

这能减少停顿。

### 6. 停顿

如果依赖实在解决不了，只能插入 bubble，等数据准备好。

其中一个很典型的场景是 `load-use hazard`：

- 前一条指令刚从内存加载数据
- 下一条立刻要用这个数据

即使有转发，有时结果来得还是太晚，仍需要停顿。

### 7. 控制冒险和分支预测

分支指令会让 CPU 面临问题：

- 下一条到底取哪条？

如果等判断结果出来再取，流水线就会空转。

所以 CPU 常用分支预测：

- 先猜
- 猜对了继续
- 猜错了回滚并重取

### 8. 为什么分支多会拖慢程序

因为分支预测失败会导致：

- 前面已经取来的指令作废
- 流水线被冲刷
- 重新取指和填充

### 9. 流水线最该记住的一句话

> 流水线的难点从来不是“并行起来”，而是“并行后如何处理冲突和依赖”。

## 十四、总线与 I/O

### 1. 总线是什么

总线可以理解为：

> 计算机内部不同部件之间共享的传输通道。

### 2. 常见总线分类

#### 地址总线

传地址信息。

#### 数据总线

传真正的数据。

#### 控制总线

传读写、中断等控制信号。

#### 一个高频点

地址总线宽度会影响理论可寻址空间上限。
例如 32 位地址在最朴素的理解下，理论上最多表示 `4GB` 地址空间。

### 3. I/O 为什么不能像寄存器一样直接处理

因为外设通常：

- 比 CPU 慢得多
- 工作方式差异很大
- 需要独立控制器协调

### 4. 设备控制器

设备控制器负责：

- 接收 CPU 命令
- 管理设备状态
- 与设备交换数据

它可以理解成 CPU 和具体外设之间的“翻译层”。
CPU 通常并不是直接理解键盘、磁盘、网卡的全部细节，而是通过设备控制器暴露出的寄存器和接口与它们交互。

### 5. 三种典型 I/O 方式

#### 轮询

CPU 不停问设备：

> 你好了吗？

缺点是 CPU 很浪费。

#### 中断

设备准备好了，再主动通知 CPU。

优点：

- CPU 不必一直空等

#### DMA

由 DMA 控制器直接在设备和内存间搬运大量数据。
CPU 只负责发起和收尾。

优点：

- 减少 CPU 参与搬运细节

#### 一句话区分

- 轮询：CPU 主动反复问
- 中断：设备好了主动通知 CPU
- DMA：大块搬运交给专门硬件，CPU 少插手

---

## 十五、中断系统

### 1. 中断是什么

中断可以理解为：

> 让 CPU 暂时停下当前工作，转去处理更紧急或更外部的事件。

### 2. 中断和异常

直觉上可以这样区分：

- **中断**：更多来自外部事件，如 I/O 完成
- **异常**：更多来自当前指令执行过程中的内部问题，如除零、缺页

再往前走一步，可以把系统调用理解成一种“程序主动请求内核服务”的受控陷入。它不是普通错误，但在很多体系结构教材里会和异常/陷入放在一起讲。

### 3. 中断处理的大致流程

1. 当前指令执行到合适边界
2. 保存现场
3. 根据中断向量找到处理程序
4. 执行中断服务程序
5. 恢复现场
6. 回到原程序继续执行

### 4. 为什么要保存现场

因为中断处理也会改寄存器和状态。
如果不保存，原程序上下文就丢了。

通常最关键的是：

- `PC`：决定回去从哪继续
- 状态字 / 标志位：决定原来的执行状态

其余通用寄存器则常由软硬件配合保存和恢复。

### 5. 为什么 I/O 很依赖中断

因为外设速度和 CPU 差异太大。
让 CPU 一直傻等设备太浪费，所以更合理的是：

- CPU 先去干别的
- 设备完成后再中断通知

---

## 十六、多核与性能指标

### 1. 主频

主频反映时钟频率。
它高通常意味着时钟周期更短，但：

> 主频高不等于性能一定高。

因为还要看：

- 每条指令多少周期
- 每周期能做多少有效工作
- 缓存和访存情况

### 2. CPI 和 IPC

#### CPI

每条指令平均需要多少时钟周期。

#### IPC

每个时钟周期平均能完成多少条指令。

两者可以互相联系理解，但不要机械背公式而不懂含义。

一个实用直觉是：

- 主频高，表示“拍子打得快”
- IPC 高，表示“每拍干的活多”
- CPI 低，表示“平均每条指令花的拍子少”

### 3. CPU 执行时间

可以粗略理解为：

> 程序执行时间 = 指令数 × 每条指令平均周期数 × 时钟周期

所以性能提升路径不止一种：

- 减少指令数
- 降低 CPI
- 提高主频

### 4. Amdahl 定律

它告诉你：

> 只优化系统中的一小部分，整体收益是有限的。

如果程序 90% 时间花在访存，你把 ALU 速度翻倍，整体提升也有限。

常见写法是：

> `总加速比 = 1 / ((1 - p) + p / s)`

其中：

- `p` 是可优化或可并行部分占比
- `s` 是这部分被加速的倍数

它最想提醒你的不是背公式，而是：优化要抓主要矛盾，系统总会被最慢、最难优化的那部分限制。

### 5. 为什么多核不等于线性加速

因为程序总有一部分：

- 不能并行
- 同步开销大
- 受共享资源限制

这也是 Amdahl 定律在多核上的常见应用。

## 十六补充、计组和并发的交界

这一节专门回答你现在这个问题：

> 锁、并发这些知识到底算不算计组？

结论是：

- **锁、线程同步、条件变量、死锁、调度策略** 更偏操作系统和并发编程
- **多核、缓存一致性、原子指令、CAS、内存屏障、伪共享** 属于计组和体系结构的交界

所以面试里如果问到锁，往往不是只想听“加锁解锁 API”，而是想看你是否理解：

- 多核下为什么会有并发问题
- 一个核心写的数据，另一个核心为什么未必立刻看见
- 为什么原子操作比普通读改写安全
- 为什么锁会有性能成本

### 1. 为什么多核会把并发问题放大

单核时代，即使有并发，很多操作更多是在时间上交替。  
多核时代则是真正可能同时发生：

- 核心 A 在改变量 `x`
- 核心 B 也在改变量 `x`

如果没有同步机制，就会发生：

- 数据竞争
- 可见性问题
- 顺序问题

### 2. 缓存一致性为什么重要

每个核心通常都有自己的 Cache。  
这很好，因为访问更快；但也带来一个问题：

> 如果核心 A 改了某个变量，核心 B 自己 Cache 里可能还留着旧值。

这就是缓存一致性问题。

现代多核处理器通常会用缓存一致性协议来保证：

- 某个核心写入后
- 其他核心最终不会一直看见过期副本

你不一定需要硬背 MESI 全状态流转，但要知道：

- 多核并发下，Cache 不只是性能工具，也会引入一致性成本
- 锁、原子操作、屏障背后都和这个有关

### 3. 原子操作到底原子在哪里

原子操作的意思是：

> 一个操作在并发观察下不可被拆分。

例如“读一个值、修改、再写回”如果用普通三步做，中间可能被别的线程插进来。  
而像 `CAS` 这种原子指令，会让这整个比较并交换过程作为一个不可分割动作完成。

### 4. CAS 是什么，为什么常被问

CAS 即 Compare-And-Swap，可以粗略理解为：

1. 先看内存中的值是不是预期值
2. 如果是，就把它改成新值
3. 如果不是，就说明期间有人改过，操作失败

它很常用于：

- 无锁数据结构
- 自旋重试
- 原子更新计数

#### 为什么 CAS 和锁经常一起被问

因为它们都在解决并发修改问题，但思路不同：

- 锁：先排他，再修改
- CAS：乐观尝试，失败就重试

#### 一个常见坑：ABA

CAS 只比较“当前值是不是 A”，但不知道它是不是经历过：

- A -> B -> A

这就是 ABA 问题。  
面试里如果被问到 CAS，顺带提一句 ABA，会很加分。

### 5. 内存可见性和内存屏障直觉

并发问题不只是“同时改同一个变量”，还有一种高频问题：

> 一个线程已经写了，另一个线程为什么还没看到？

原因可能包括：

- 数据还在寄存器或本地 Cache
- 指令和内存访问被重排序

所以需要一些同步语义来约束：

- 哪些写必须在别的线程可见前完成
- 哪些读不能越过某些写

内存屏障可以理解成：

> 在某些关键点，约束编译器和 CPU 不要把内存操作随便乱换顺序。

学习阶段你不用背所有屏障分类，但至少要知道：

- 并发正确性和“执行顺序”有关
- CPU 为了性能可能会重排
- 锁、原子操作、语言内存模型常会借助屏障保证正确性

### 6. 为什么伪共享属于计组和并发交界

你前面已经学过伪共享，它本质上是：

- 多线程程序逻辑上没共享同一个变量
- 但硬件上共享同一个 Cache line

所以它同时涉及：

- 并发程序结构
- Cache line
- 缓存一致性

这就是为什么很多“锁竞争”看起来像并发问题，根子却能追到硬件缓存层。

### 7. 这一节该带走什么

如果面试官问并发锁，你至少要能说出：

1. 多核 + 各核心私有 Cache 会带来一致性与可见性问题
2. 原子操作和 CAS 的基础来自硬件提供的不可分割指令语义
3. 内存屏障和同步原语在约束重排、保证可见性
4. 伪共享说明并发性能问题常常能追到 Cache line 这一层

如果要系统学线程、锁、死锁、条件变量、调度、线程池，那已经更属于一份独立的“操作系统与并发”文档了。

## 十七、经典综合题

### 1. 为什么顺序访问数组更快

因为顺序访问更符合空间局部性：

- 相邻元素常在同一个或相邻 Cache line
- 一次调入后，后续访问容易命中

而随机访问更容易频繁 miss。

### 2. 从 `a + b` 到 CPU 执行发生了什么

一个高频回答思路是：

1. 编译器先把高级语言翻译成机器指令
2. CPU 根据 `PC` 取指
3. 控制器译码
4. 需要的数据从寄存器或内存取到执行单元
5. ALU 做加法
6. 结果写回寄存器或内存
7. 如果数据不在 Cache，可能触发更慢的访存

### 3. 为什么 Cache miss 很贵

因为 miss 往往意味着：

- CPU 要停下来等更慢层级
- 可能一路 miss 到主存
- 还可能带来替换、写回等额外开销

### 4. 为什么分支多会影响流水线性能

因为分支让“下一条该取哪条指令”变得不确定。
如果预测错了，流水线前面已经取来的内容可能全部作废。

### 5. 为什么局部性几乎无处不在

因为：

- Cache 利用局部性
- TLB 利用局部性
- 指令执行也有局部性
- 很多性能优化本质上都是在提升局部性

## 十八、容易混淆和高频误区

### 1. 补码和反码不要混

真正工程里最重要的是补码。
反码更多是理解补码推导过程。

### 2. Cache 命中率高不等于程序一定快

还要看：

- miss 的代价多大
- 是否有分支预测失败
- 是否有流水线停顿
- 是否有 TLB miss

### 3. 主频高不等于性能高

因为性能还受 CPI、IPC、访存延迟、并行度影响。

### 4. 流水线提升吞吐，不等于单条指令延迟一定下降

这两个概念经常被混淆。

### 5. 多核不等于单线程更快

多核提升的是可并行任务的总吞吐，不一定提升单线程性能。

### 6. TLB、页表、Cache 别混

- TLB 缓存地址翻译
- Cache 缓存数据/指令
- 页表保存完整映射关系

## 十九、面试速背提纲

### 1. 计组在学什么

- 程序如何被 CPU 执行
- 数据如何表示
- 存储系统如何分层
- 为什么性能瓶颈经常出在访存而不是算术

### 2. 数据表示

- 整数重点是补码
- 浮点数重点是符号位、指数、尾数和精度问题
- 大小端和对齐是工程常见点

### 3. 存储系统

- 寄存器最快，离 CPU 最近
- Cache 利用局部性减少主存访问
- 主存大但慢
- 外存更大更慢

### 4. Cache

- 映射方式：直接映射、组相联、全相联
- 替换策略：LRU、FIFO、随机
- 写策略：写直达、写回
- miss 成本通常很高

### 5. CPU

- 关键部件：控制器、运算器、寄存器、ALU
- 关键寄存器：PC、IR、MAR、MDR、PSW

### 6. 指令执行流程

- 取指、译码、执行、访存、写回

### 7. 流水线

- 提升吞吐率
- 三类冒险：结构、数据、控制
- 解决手段：转发、停顿、分支预测

### 8. I/O

- 轮询浪费 CPU
- 中断更高效
- DMA 适合大块数据搬运

### 9. 性能指标

- 主频、CPI、IPC、执行时间
- Amdahl 定律说明局部优化收益有限

### 10. 综合题主线

- 局部性 -> Cache
- 取指译码执行 -> CPU
- 分支 -> 流水线
- 缺页 / TLB / Cache -> 访存链路
