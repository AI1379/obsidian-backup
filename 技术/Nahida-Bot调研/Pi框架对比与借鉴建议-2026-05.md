## 二、Agent Loop 核心对比

### nahida-bot: `AgentLoop` (OOP, 1042 行)

```python
class AgentLoop:
    async def run_stream(self, ...):
        for step in range(1, max_steps + 1):
            # 1. 构建上下文
            prompt_messages = builder.build_context(...)
            # 2. 调用 provider
            response = await self._call_provider_with_retry(...)
            # 3. 检查 tool calls
            if not response.tool_calls:
                yield done_event; return
            # 4. 串行执行工具
            for tool_call in response.tool_calls:
                result = await self._execute_tool_with_lifecycle(...)
            # 5. 继续循环
```

**特点:**
- 步数限制 (`max_steps=8`)
- 串行工具执行 (for 循环逐个执行)
- 带重试的 provider 调用
- 详细的验证和日志
- `stop_event` 支持取消

### Pi: `agentLoop()` (函数式, ~500 行)

```typescript
function runLoop(context, config, signal, emit, streamFn):
    while (true):  // 外层循环处理 steering + follow-up
        while (hasMoreToolCalls || pendingMessages.length > 0):
            // 1. 注入 steering 消息
            // 2. 流式获取 assistant 响应
            // 3. 并行执行工具 (Promise.all)
            // 4. prepareNextTurn 钩子
            // 5. 检查 shouldStopAfterTurn
            // 6. 检查新的 steering 消息
        // 检查 follow-up 消息
```

**特点:**
- 无限循环 (靠 hooks 终止)
- 并行工具执行 (默认 `Promise.all`)
- Steering 机制 (中途注入用户消息)
- Follow-up 机制 (agent 完成后继续)
- `beforeToolCall` / `afterToolCall` 钩子
- `prepareNextTurn` 钩子 (动态切换模型/thinking level)
- `AbortSignal` 取消

## 三、关键差异分析

### 🔥 1. 工具执行: 串行 vs 并行

**nahida-bot** — 串行:
```python
# agent/loop.py, _execute_tools()
for tool_call in response.tool_calls:
    result = await self._execute_tool_with_lifecycle(tool_call, ...)
```

**Pi** — 默认并行:
```typescript
// agent-loop.ts, executeToolCallsParallel()
const orderedFinalizedCalls = await Promise.all(
    finalizedCalls.map(entry => typeof entry === "function" ? entry() : ...)
);
```

**影响**: 当模型一次调用多个工具 (如同时读两个文件), nahida-bot 串行等待, Pi 并行完成。
这是一个**低难度高收益**的改进点。

**建议**: 在 `AgentLoop._execute_tools()` 中改为 `asyncio.gather()` 并行执行。

---

### 🔥 2. Steering (中途干预)

**nahida-bot**: ❌ 无。用户必须等 agent 完全回复后才能发下一条消息。

**Pi**: ✅ 有 `getSteeringMessages()` 和 `getFollowUpMessages()` 钩子。

```typescript
// Pi 的 steering 机制
pendingMessages = (await config.getSteeringMessages?.()) || [];
// 在下一轮循环中注入这些消息
```

**影响**: 在长任务 (如 arXiv 摘要处理) 中, 用户无法中途纠正方向。

**建议**: 中等难度。需要:
1. 在 `SessionRunner` 中维护一个 `pending_messages: asyncio.Queue`
2. 在 `AgentLoop.run_stream()` 的 step 循环中检查队列
3. Gateway 接收到新消息时放入队列而不是拒绝

---

### 🔥 3. 会话树 vs 扁平列表

**nahida-bot**: 扁平消息列表 (`list[ContextMessage]`), 支持上下文压缩 (DANGER_ZONE 协议)。

**Pi**: 树状结构 (`SessionTreeEntry`), 每个 entry 有 `parentId`, 支持分支、回滚。

```typescript
// Pi 的 Session tree
interface SessionTreeEntry {
    id: string;
    parentId: string | null;
    type: "message" | "compaction" | "branch_summary" | "custom_message" | ...
}
// 支持从任意节点创建分支
// 支持在不同分支间切换
```

**影响**: 这是两个项目最根本的架构差异。Pi 的树结构:
- 天然支持"撤回重来"
- 天然支持上下文压缩 (compaction 就是树的一个特殊 entry)
- 天然支持对话分支

nahida-bot 的扁平结构:
- 简单直接
- 但 DANGER_ZONE 压缩是破坏性的 (无法回滚)
- 上下文溢出需要手动处理

**建议**: 极高难度。改造会话存储层几乎等于重写。但可以参考 Pi 的 compaction 思路改善现有压缩。

---

### 🔥 4. 工具生命周期钩子

**nahida-bot**: ❌ 无 before/after 钩子。工具直接执行。

**Pi**: ✅ 完整的钩子系统:

```typescript
interface AgentLoopConfig {
    beforeToolCall?: (ctx, signal) => Promise<{ block?: boolean; reason?: string }>
    afterToolCall?: (ctx, signal) => Promise<{ content?; details?; isError?; terminate? }>
}
```

**影响**: 钩子可以实现:
- **权限控制**: beforeToolCall 检查敏感操作
- **审计日志**: 记录每次工具调用
- **结果过滤**: afterToolCall 过滤敏感输出
- **提前终止**: terminate 标志

**建议**: 低难度。在 `AgentLoop._execute_tool_with_lifecycle()` 中加入 before/after 回调。

---

### 🔥 5. Compaction (上下文压缩) — 最精妙的设计

**nahida-bot**: 基于字符/token 预算裁剪旧消息, 写入 working-buffer.md。

**Pi**: 基于 LLM 的智能摘要压缩, 极其精细:

1. **Token 预算计算**: 使用 provider 返回的真实 usage + 尾部估算
2. **切割点选择**: 智能选择 turn 边界 (不会切断一个对话轮次)
3. **增量摘要**: 如果有上一次 compaction, 用更新摘要而非重新生成
4. **Turn 拆分**: 如果一个 turn 太大, 会分别摘要前缀和保留后缀
5. **文件操作追踪**: 追踪 compaction 区间内读/写过的文件, 附在摘要里
6. **结构化摘要模板**: Goal / Progress / Decisions / Next Steps / Critical Context

**建议**: 高难度但极高价值。可以分阶段:
- P0: 借鉴 Pi 的结构化摘要模板
- P1: 实现增量摘要
- P2: 实现 turn-aware 切割

---

### 🔥 6. Skill 系统

**nahida-bot**: 手动维护的 Markdown 文件 (AGENTS.md 中硬编码引用)。

**Pi**: 标准化的 Skill 发现和加载系统:

```typescript
interface Skill {
    name: string;           // 必须匹配目录名
    description: string;    // 必填, 用于模型选择
    content: string;        // Markdown 内容
    filePath: string;       // 绝对路径
    disableModelInvocation?: boolean;
}
```

加载规则:
1. 递归扫描目录
2. 优先加载 `SKILL.md` 文件
3. 支持 frontmatter (YAML)
4. 遵守 `.gitignore` / `.ignore` 规则
5. 验证 name 格式 (`^[a-z0-9-]+$`)
6. 遵循 [agentskills.io](https://agentskills.io) 规范

**建议**: 中等难度。nahida-bot 可以:
1. 统一 skill 格式为 `SKILL.md` + frontmatter
2. 自动扫描 `skills/` 目录
3. 把 skill 信息注入 system prompt

## 六、具体实现建议

### P0: 工具并行执行

在 `nahida_bot/agent/loop.py` 的 `_execute_tools` 方法中:

```python
# 当前: 串行
for tool_call in response.tool_calls:
    result = await self._execute_tool_with_lifecycle(tool_call, ...)
    messages.append(self._build_tool_message(...))

# 改为: 并行
async def _execute_tools(self, ...):
    tasks = [self._execute_tool_with_lifecycle(tc, ...) for tc in response.tool_calls]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    for tool_call, (result, attempt, phase) in zip(response.tool_calls, results):
        if isinstance(result, Exception):
            result = ToolExecutionResult.error(...)
        messages.append(self._build_tool_message(...))
```

### P0: before/after 钩子

在 `AgentLoopConfig` 中新增:

```python
@dataclass
class ToolHookContext:
    tool_call: ToolCall
    args: dict
    message: ContextMessage | None = None

class AgentLoopConfig:
    before_tool_call: Callable[[ToolHookContext], Awaitable[bool | None]] | None = None
    after_tool_call: Callable[[ToolHookContext, ToolExecutionResult], Awaitable[ToolExecutionResult | None]] | None = None
```

### P1: Skill 标准化

参考 Pi 的 `skills.ts`, 在 nahida-bot 中:
1. 定义 `SKILL.md` + YAML frontmatter 格式
2. 启动时自动扫描 `skills/` 目录
3. 将 skill 描述注入 system prompt
4. 支持按需加载 (不在每次请求中都加载所有 skill 内容)

## 七、总结

Pi 和 nahida-bot 是**完全不同层次**的项目:

- **Pi** 是一个精心打磨的 **Agent 引擎内核** — 它的 agent loop 设计质量非常高,
  特别是并行执行、生命周期钩子、steering、会话树和 compaction 这些。
  但它不含任何"让 Agent 活起来"的东西。

- **nahida-bot** 是一个 **完整的 AI 伴侣运行时** — 有记忆、有人格、有 IM、
  有定时任务、有子代理。这些是 Pi 完全没有的。

最值得从 Pi 借鉴的是它的 **Agent Loop 内核设计** — 工具并行、钩子系统、
compaction 策略。这些是 nahida-bot 当前的短板, 而且改进难度不高 (除了会话树)。

一句话: **Pi 的引擎做得好, nahida-bot 的车造得全。把 Pi 的引擎技术移植过来, nahida-bot 会更强。**
