## 2. nahida-bot 的现实处境

### 2.1 独特优势（来自旅行者的真实背景）

1. **数学/信息与计算科学专业** — 浙大数学学院，算法基础扎实
2. **RAG / 知识图谱研究深度** — Obsidian 笔记库中有大量相关论文整理：
   - GAM: Hierarchical Graph-based Agentic Memory
   - PolyG: Adaptive Graph Traversal for Diverse GraphRAG Questions
   - Graph RAG (Microsoft)
   - RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval
   - HIRAG, MacRAG 等
3. **Agentic Search 研究方向** — 已有研究综述笔记
4. **学术工作流实践者** — Zotero、arXiv digest、论文追踪已在日常使用
5. **真实用户场景** — 自己就是 nahida-bot 的目标用户

### 2.2 竞品的盲区

| 竞品 | 记忆/知识系统 | 问题 |
|:---|:---|:---|
| NanoBot | Dream two-stage memory | 向量检索 + 摘要，**无知识图谱** |
| Hermes | FTS5 + Honcho | 文本搜索 + 用户建模，**无结构化知识** |
| AstrBot | RAG | 传统向量RAG，**无推理能力** |
| OpenClaw | 文件式 + LanceDB | 简单向量搜索，**无关系推理** |
| Khoj | 文档RAG | **最强RAG**，但不是 Agent 框架 |

**所有竞品的知识系统都是「flat」的 — 扁平的向量/文本检索，没有结构化的知识图谱推理能力。**

## 3. 差异化方向：Knowledge-Native Agent

### 3.1 核心定位

> **nahida-bot: The Knowledge-Native Agent Framework**
> 
> 不是"又一个 Python OpenClaw"，而是"**第一个把知识图谱作为记忆一等公民的 Agent 框架**"。

### 3.2 为什么这个方向可行

1. **技术壁垒高** — Graph RAG 不是调个 API 就能做的，需要算法深度
2. **竞品不会做** — NanoBot 在拼渠道、Hermes 在拼自我学习、AstrBot 在拼插件生态，没人专注知识层
3. **你有理论基础** — GAM 论文的 hierarchical graph-based agentic memory 就是现成的架构参考
4. **市场需求真实** — 学术研究者、知识工作者、学生群体都需要"能理解知识关系的 AI"

### 3.3 三层差异化

```
┌─────────────────────────────────────────────────┐
│  Layer 3: 知识工作流（最外层，用户可见）           │
│  论文追踪 / Zotero 集成 / arXiv digest / 笔记同步  │
├─────────────────────────────────────────────────┤
│  Layer 2: 知识推理（中间层，核心竞争力）           │
│  Graph RAG / 知识图谱记忆 / 跨文档推理 / 关系发现   │
├─────────────────────────────────────────────────┤
│  Layer 1: Agent 基座（最内层，必要但非差异化）      │
│  Agent Loop / Workspace / Plugin / Channel       │
└─────────────────────────────────────────────────┘
```

**Layer 1 是所有人都在做的。Layer 2 和 Layer 3 才是 nahida-bot 的战场。**

## 4. 具体落地路径

### 4.1 Layer 2: 知识图谱记忆系统

#### 4.1.1 架构参考：GAM (Hierarchical Graph-based Agentic Memory)

你的 Obsidian 笔记库中已有这篇论文的整理。核心思路：

```
传统记忆:    [对话1] [对话2] [对话3] → 向量化 → 余弦相似度检索
                                              ↓
                                          flat, 无结构

GAM 式记忆:  [对话1] [对话2] [对话3] → 实体抽取 → 知识图谱
                                              ↓
                                    [人] --关系--> [事件] --关系--> [概念]
                                              ↓
                                     图遍历 + 子图检索 + 推理
```

#### 4.1.2 记忆三层架构

```python
# 记忆系统设计
class MemoryLayer(Enum):
    EPISODIC = "episodic"     # 短期：最近 N 轮对话（现有 SQLite）
    SEMANTIC = "semantic"     # 中期：LLM 摘要压缩（现有规划）
    KNOWLEDGE = "knowledge"   # 长期：知识图谱（新差异化）
```

| 层级 | 存储 | 检索方式 | 生命周期 |
|:---|:---|:---|:---|
| Episodic | SQLite | 时间窗口 + session ID | 自动淘汰 |
| Semantic | SQLite + embedding | 向量相似度 | 定期摘要合并 |
| Knowledge | 图数据库 (SQLite + 关系表) | 图遍历 + 子图匹配 | 永久，持续丰富 |

#### 4.1.3 知识图谱记忆的 API 设计

```python
class KnowledgeGraphMemory:
    """知识图谱记忆 — nahida-bot 的核心差异化模块。"""
    
    async def extract_and_store(self, text: str, source: str) -> list[Entity]:
        """从文本中抽取实体和关系，存入知识图谱。"""
        # 1. LLM 抽取实体（人、事件、概念、偏好、决定...）
        # 2. LLM 抽取关系（喜欢、属于、发生在...、导致...）
        # 3. 合并到现有图谱（实体消歧、关系去重）
        ...
    
    async def query(self, question: str) -> SubGraph:
        """基于问题检索相关知识子图。"""
        # 1. 从问题中识别关键实体
        # 2. 图遍历扩展到相关实体（BFS/DFS，N跳）
        # 3. 返回子图用于上下文构建
        ...
    
    async def reason(self, query: str, subgraph: SubGraph) -> str:
        """在知识子图上做推理。"""
        # 将子图序列化为文本 + 交给 LLM 推理
        ...
    
    async def merge(self, new_facts: list[Fact]) -> MergeReport:
        """合并新事实到图谱，处理冲突。"""
        # 1. 实体对齐（"RM" vs "Renatus" vs "旅行者"）
        # 2. 关系更新（偏好变化、过时信息淘汰）
        # 3. 返回合并报告
        ...
```

#### 4.1.4 为什么不用纯向量RAG？

| 维度 | 向量 RAG | Graph RAG |
|:---|:---|:---|
| "旅行者喜欢什么游戏？" | 搜索所有提到游戏的片段，可能漏 | 直接查 `(旅行者)-[喜欢]->(游戏)` |
| "最近有什么变化？" | 无法回答 | 查时间衰减的边权重 |
| "A 和 B 有什么关系？" | 不擅长多跳推理 | 图遍历自然支持 |
| "之前做的决定现在怎样了？" | 难以追踪 | 实体 + 事件 + 时间线 |
| 近义词/模糊匹配 | ✅ 强 | ⚠️ 需要额外处理 |

**结论：两者互补。Graph RAG 解决结构化推理，向量检索解决模糊匹配。nahida-bot 应该两者都有。**

### 4.2 Layer 2+: Agentic Search 集成

你已有的 Agentic Search 研究综述可以直接产品化：

- Agent 在回答问题时自主决定是否需要检索
- 多步推理 + 多次检索的闭环
- 检索源扩展：arXiv、Google Scholar、本地知识库

### 4.3 Layer 3: 学术工作流原生支持

#### 4.3.1 一等公民工具

```yaml
# 内置学术工具插件
tools:
  - arxiv_search        # arXiv 论文搜索
  - zotero_manage       # Zotero 文献管理（添加/搜索/导出）
  - paper_summarize     # 论文摘要（PDF 解析 + LLM 总结）
  - citation_graph      # 引用关系图构建
  - knowledge_sync      # 知识图谱 ↔ Obsidian/Zotero 同步
  - daily_digest        # 每日论文/arXiv 推送
```

#### 4.3.2 知识工作流

```
论文发现 → 自动抽取实体/关系 → 存入知识图谱
                ↓
        与已有知识关联/对比
                ↓
    Obsidian 笔记自动生成/更新
                ↓
    定期知识图谱可视化（"你的知识地图"）
```

### 4.4 技术选型建议

| 组件 | 推荐方案 | 理由 |
|:---|:---|:---|
| 图存储 | SQLite + 关系表（轻量）或 NetworkX | 不引入额外依赖，SQLite 已有 |
| 实体抽取 | LLM-based (few-shot prompt) | 不需要训练 NER 模型 |
| 向量检索 | LanceDB 或 ChromaDB | 轻量、嵌入式、Python 原生 |
| PDF 解析 | PyMuPDF + LLM | 开源、快速 |
| 图可视化 | pyvis / networkx → HTML | WebUI 中展示知识图谱 |

## 5. 不要做的事情

### ❌ 不拼渠道数量

NanoBot 已有 12+ 渠道，AstrBot 有 7+ + 1000 插件。从 1 个 Telegram 去追赶是浪费精力。

### ❌ 不拼通用插件生态

AstrBot 的 1000+ 社区插件是时间积累的结果，无法短期复制。

### ❌ 不做"更轻量的 OpenClaw"

NanoBot 已经占据了这个定位（"Ultra-Lightweight Personal AI Agent"）。

### ❌ 不做自我学习（短期）

Hermes Agent 的自我学习闭环是它的核心，但实现复杂度极高。等 nahida-bot 的知识图谱稳定后再考虑。

### ❌ 不拼部署便捷性

pipx 一键安装足够了，不需要 Docker/serverless 等。

## 6. 修订后的优先级路线图

### Phase 0-4: 保持现有计划不变

核心 Agent 框架必须扎实，这是地基。

### Phase 4.5（新增）：Knowledge Layer 原型 ⭐

> 这应该成为 v0.2 的核心卖点。

- [ ] 实体抽取引擎（LLM few-shot prompt）
- [ ] 知识图谱存储（SQLite 关系表 + 实体/关系模型）
- [ ] 图检索 API（子图提取、N跳遍历）
- [ ] 与现有记忆系统集成（KnowledgeGraphMemory 层）
- [ ] 基础图可视化（WebUI 或 HTML 导出）

### Phase 5: 保持 Gateway-Node（但降低优先级）

分布式执行是"nice to have"，不是差异化。

### Phase 6: 学术工作流工具包 ⭐

- [ ] arXiv 搜索工具
- [ ] Zotero API 集成
- [ ] PDF 论文解析 + 摘要
- [ ] 知识图谱 ↔ Obsidian 同步
- [ ] 每日论文 digest 推送

### Phase 7: WebUI（知识图谱可视化为重点）

- [ ] 知识图谱交互式浏览
- [ ] 记忆时间线视图
- [ ] 实体/关系搜索界面

## 7. 竞品不会做的事

| 事情 | 为什么竞品不会做 | 为什么 nahida-bot 能做 |
|:---|:---|:---|
| Graph RAG 记忆 | 算法复杂、投入大、不是"聊天机器人"的核心需求 | 你有 GAM 等论文的理论基础，且有真实使用场景 |
| 学术工作流集成 | 目标用户是"所有人"，不是"研究者" | 你自己就是目标用户，dogfooding |
| 知识图谱可视化 | 需要前端投入，ROI 不明确 | Python 生态有 pyvis 等，轻量实现即可 |
| 跨文档推理 | 需要 Graph RAG 基础 | 有了知识图谱层自然支持 |
